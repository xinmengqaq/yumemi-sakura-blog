## 1. 业务目标与完成定义

本计划交付个人博客前台首期的三个真实公开页面：

- `/`：首页，展示后端返回的精选文章和最新文章，并实现樱花车站叙事氛围。
- `/articles`：文章列表页，支持关键词、分类、年月、多标签和分页筛选，筛选状态可从 URL 恢复。
- `/articles/:id`：文章详情页，完整渲染公开正文、目录、阅读进度、阅读偏好、代码复制和匿名点赞。

整体完成必须同时满足：

- 三页以真实浏览器视口重新组织布局：原型只参考视觉层级、内容节奏和组件关系，不还原原型比例、不把原型图片像素当 CSS 像素，也不按原型组件尺寸逐块照搬。
- 宽桌面必须根据实际浏览器宽度增加页面两侧留白，使用可读性约束的内容容器、正文宽度和栏间距；组件尺寸、排列和留白随视口重新计算。
- 所有文章、分类、标签、归档和统计数据来自 Spring Boot 已实现的公开接口；不使用 `/admin/**`，不造假数据，不在前端过滤草稿或隐藏文章。
- 首页、文章列表、文章详情分别拥有独立页面结构；不得把同一种文章列表组件硬套到三个页面。
- 已设计功能必须完整实现加载、空数据、失败、重试、键盘、移动端和减少动态效果状态，不得保留占位业务模块。
- 后端工程 `D:/daimai/项目学习/springboot` 不做代码修改；若前端联调发现契约冲突，先停止并升级确认，不以修改后端实现迁就前端。
- 页面文件只做页面级编排；可复用视觉、数据、状态和行为抽到职责单一的组件、Hook、类型或工具中，不新增推测性抽象。
- 正式交付前通过 BDD/TDD 门禁、前端全量质量命令和真实浏览器视觉验收。

### 3.2 当前前端事实

- React + TypeScript + Vite，使用 React Router、TanStack Query、Axios、Zustand、Lucide、Vitest 和 Testing Library。
- `FrontLayout` 仅有空壳容器，`HomeView` 仍是“等待后端公开接口”的占位页；尚无公开文章 API、Query、类型和文章列表/详情路由。
- `request` 的 `baseURL` 已是 `/api`，因此前台 API 封装使用 `/home`、`/articles` 等相对路径，禁止再次写 `/api/home` 形成重复前缀。
- 现有 `article.ts`、`taxonomy.ts` 是后台 `/admin/**` 契约，不能作为前台公开接口复用；可复用的是 `request`、`storage`、基础 `ApiResult/PageResult` 和通用 UI 组件中不带后台语义的部分。
- 当前 `ApiError` 不保留后端错误响应的 `data`，不足以读取点赞限频的 `retryAfterSeconds`，需在共享请求层以向后兼容方式补充结构化详情。
- 当前未安装 GSAP。只有复杂的滚动入场、页面编排和统一动画生命周期需要新增 `gsap`、`@gsap/react`；简单 hover、焦点和显隐继续使用 CSS。

### 3.3 后端公开契约事实

| 能力 | 方法与路径 | 前端用途 |
|---|---|---|
| 首页 | `GET /api/home` | 精选与最新文章，后端决定数量、顺序和公开过滤 |
| 文章分页 | `GET /api/articles` | `page/size/keyword/categoryId/tagIds/year/month` 组合筛选 |
| 筛选元数据 | `GET /api/articles/meta` | 只生成存在文章的年月选项 |
| 文章详情 | `GET /api/articles/{id}` | 正文、标签和递增后的浏览量 |
| 分类 | `GET /api/categories` | 导航分类，后端已按可见性和顺序过滤 |
| 标签 | `GET /api/tags` | 列表页多标签筛选 |
| 点赞 | `POST /api/articles/{id}/like` | 匿名一次性点赞，返回最新 `likeCount` |

关键约束：

- `tagIds` 在 HTTP 查询中使用逗号分隔，后端按 OR 语义处理；前端进入 URL、Query Key 和请求前均去重并按数字升序排列。
- `month` 存在时必须同时发送 `year`。
- 详情不存在、草稿或隐藏统一表现为不可公开访问，不渲染旧缓存正文。
- 重复点赞返回业务码 `409`；限频返回业务码 `429`，`data.retryAfterSeconds` 提供等待秒数。
- 匿名访客识别、防重复和 IP 限频均由后端 HttpOnly Cookie 与服务端记录完成，前端不得自建安全判断。

### 3.4 外部技术依据

- GSAP React：`https://gsap.com/resources/React`，查询日期 2026-07-12。React 中使用 `useGSAP`、组件作用域和自动清理。
- GSAP ScrollTrigger：`https://gsap.com/docs/v3/Plugins/ScrollTrigger/`，查询日期 2026-07-12。滚动触发动画必须注册插件、按页面顺序创建并在动态内容改变布局后刷新。
- GSAP matchMedia：`https://gsap.com/docs/v3/GSAP/gsap.matchMedia/`，查询日期 2026-07-12。响应式动画和 `prefers-reduced-motion` 由同一生命周期管理。

目录是职责边界，不要求预先创建空文件。只有 Task 实际需要时才创建对应文件。

### 4.2 文件维护约束

- `HomeView`、`ArticleListView`、`ArticleDetailView` 只组合页面区块、Query 状态和路由参数，不容纳正文解析、筛选编解码、花瓣引擎或点赞错误解释。
- 一个组件只负责一种可命名行为；同一页面区块出现独立状态、独立测试或独立响应式规则时拆成子组件。
- 跨两处以上且视觉与业务语义一致时才抽取复用；首页精选布局与文章列表交错布局保持独立，不为减少少量 JSX 强行统一。
- CSS 按 `front-shared.css`、页面 CSS 和复杂组件 CSS 分层；禁止把三页全部样式继续堆入 `global.css` 或单个 `front.css`。
- 常量集中但不过度配置化：分页大小、断点、动画时长、花瓣密度进入明确模块或 CSS 变量；一次性布局数值留在所属页面样式。
- 新增依赖仅限 `gsap`、`@gsap/react`；不得再引入 Markdown 渲染器、轮播库、弹窗库或粒子库，现有 `unified/remark/DOMPurify/Floating UI/Lucide` 足以覆盖需求。

### 4.3 数据流

```text
URL/路由参数
    ↓ 规范化
页面 Query Hook → src/api/publicContent.ts → /api 公开接口
    ↓
页面独立状态 + 可复用展示组件
    ↓
用户筛选/点赞/阅读设置
    ├─ 筛选：写回 URL，驱动 Query Key 与请求
    ├─ 点赞：Mutation 成功后更新详情并失效相关列表缓存
    └─ 阅读设置：写入 storage，仅影响当前设备
```

### 4.4 动画边界

- GSAP 只用于需要运行时控制或滚动生命周期的动画：列表一次性入场、页面切换编排、首页单会话列车和必要视差。
- 全屏花瓣优先使用单个 `canvas` 或受控 DOM 粒子层；无论实现方式，必须统一暂停、降密度、销毁和减少动态效果逻辑，不引入第三方粒子库。
- 阅读进度使用原生滚动测量与 `requestAnimationFrame` 节流，同一数值同时驱动顶部进度线和右栏百分比，不创建两套 ScrollTrigger。
- 动画只改变 `transform` 和 `opacity` 等合成属性；不持续动画 `top/left/width/height`，不对所有节点滥用 `will-change`。
- 所有 GSAP 逻辑使用 `useGSAP`、组件 ref 作用域和清理；通过 `gsap.matchMedia()` 同时处理桌面、移动端与 `prefers-reduced-motion`。

## 5. 跨层契约与业务不变量

1. 前台永远不调用 `/admin/**`，也不携带管理员能力作为页面可用性的前提。
2. 后端返回顺序就是展示顺序；前端不重排精选、最新、分类、标签和文章列表。
3. `isRecommend` 只映射“精选”，`isTop` 只映射“置顶”，不得互换或通过位置猜测。
4. 筛选 URL 是文章列表状态的唯一可恢复来源；刷新、前进、后退、复制链接后结果一致。
5. 任一筛选变化使 `page=1`；只有分页操作保留其他筛选并改变页码。
6. 分类仅在导航“文章”子菜单选择；列表筛选区不得重复分类控件。
7. 图片 URL 变化必须清除旧错误状态并重新加载；只有缺失或真实失败时显示中性占位。
8. 点赞请求中不乐观增加数量；成功使用服务端返回值，失败保持原值。
9. 详情切换文章 ID 时不得短暂显示上一文章正文；不可公开访问时只显示统一错误状态。
10. 阅读偏好不依赖登录，不上传后端；损坏的本地数据回退默认值。
11. 评论数只作为统计数字，不出现任何评论模块或“开发中”占位。
12. 原型中的图片是视觉方向，不是正式内容资产；接口无图时使用设计规定的灰色占位。

## 6. 风险、防御与恢复

| 风险 | 发生条件与影响 | 预防与检测 | 处置与恢复 | 级别 |
|---|---|---|---|---|
| 公开/后台接口混用 | 复用后台 API 文件导致权限泄露或 401 | 独立 `publicContent` API 与 Query；测试断言无 `/admin/` | 阻断交付并修正调用链 | P0 |
| URL 与 Query Key 不一致 | 标签顺序、非法参数或回退状态不同 | 单一规范化工具；表驱动测试 | 丢弃非法项并 `replace` 为规范 URL | P0 |
| 点赞限频信息丢失 | 请求层未保留响应 `data` | 扩展 `ApiError.details` 并测试 409/429 | 普通错误降级为通用提示，计数不变 | P0 |
| 旧正文泄露 | 详情切换或 404 时保留旧缓存 | Query Key 含 ID；加载/错误分支不渲染旧 data | 清空当前渲染上下文，提供返回/重试 | P0 |
| XSS | 后端 Markdown/HTML 含恶意内容 | 现有 unified + DOMPurify 白名单渲染；危险协议测试 | 拒绝危险节点/属性，不使用 `dangerouslySetInnerHTML` 直出未净化内容 | P0 |
| 动画妨碍阅读 | 高密花瓣、重复入场、低端设备掉帧 | 减少动态效果、移动降密度、页面隐藏暂停、性能检查 | 自动禁用非关键动画，保留完整静态内容 | P1 |
| 图片布局跳动 | 图片晚加载或失败 | 固定 `aspect-ratio`、尺寸约束和统一占位 | 单图失败局部降级 | P1 |
| 长目录覆盖正文 | H2/H3 很多或标题很长 | 右栏固定宽度、目录内部滚动、文本换行 | 移动端进入底部抽屉 | P1 |
| 单文件继续膨胀 | 页面把所有行为和样式集中实现 | 页面编排边界、独立测试边界和审查清单 | 在同一 Task 内按职责拆分，不做无关重构 | P1 |
| 后端不可用 | 本地联调或生产网络失败 | 各 Query 独立错误态和重试 | 保留可用区域与筛选，用户可重试 | P1 |
| 后端契约冲突 | 实际字段、错误码或路径与设计不同 | 联调前契约测试和 Swagger/源码核对 | 停止相关 Task，记录差异并请求决策；不改后端 | P0 |

不适用：本计划不修改数据库、事务、缓存、消息队列、部署拓扑或后端限流实现，因此不规划数据库迁移、服务端回滚和容量基础设施。

### Task 2：建立公开 Query 缓存与筛选规范化能力

**目标结果：**
公开首页、列表、详情、分类、标签、归档和点赞拥有相互隔离、可预测的 TanStack Query 契约。

**依据与上下文入口：**

- `[DESIGN §12]` Query Key 与缓存更新规则。
- `src/queries/article.ts` 现有模式。

**范围与边界：**

- 必须完成：公开 Query Keys、Query Hooks、点赞 Mutation、筛选规范化工具和测试。
- 不得实施：页面 UI、URL 控件、阅读设置。
- 允许修改：`src/queries/publicContent.ts`、`src/utils/publicArticleFilters.ts` 及测试。
- 必须保持：后台 `articleQueryKeys` 不受公开缓存失效影响。

**契约与约束：**

- Query Key 含全部规范化筛选参数；`tagIds` 去重升序。
- 点赞成功直接写入当前详情 `likeCount`，并失效首页和公开列表；失败不写缓存。
