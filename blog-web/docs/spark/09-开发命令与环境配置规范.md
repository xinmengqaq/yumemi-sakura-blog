# 09-开发命令与环境配置规范

## 设计目标

开发命令和环境配置用于保证前端项目可以稳定启动、联调、测试和构建。

本项目需要和 Spring Boot 后端配合开发，因此前端环境配置必须清楚说明：

- 前端服务如何启动。
- 后端接口如何代理。
- 环境变量如何命名。
- 本地开发和生产构建如何区分。
- 测试、类型检查、代码检查、构建命令如何统一。
- 前端和后端联调时需要注意什么。

## Node.js 版本

前端项目使用 Node.js LTS 版本。

建议使用 Node.js 20 或更新的 LTS 版本。

不要使用过旧 Node.js 版本，避免 Vite、React、TypeScript 工具链出现兼容问题。

## 包管理器

前端项目使用 npm。

原因：

- npm 随 Node.js 默认安装。
- 使用门槛低。
- 官方资料多。
- 当前项目不需要 monorepo 包管理能力。

如果后续需要切换 pnpm，应先补充规格并统一迁移锁文件。

## 常用命令

前端项目应提供以下 npm scripts：

```json
{
  "scripts": {
    "dev": "vite",
    "build": "tsc -b && vite build",
    "preview": "vite preview",
    "test": "vitest",
    "test:run": "vitest run",
    "lint": "eslint .",
    "typecheck": "tsc --noEmit"
  }
}
```

如果后续加入格式化检查，可以补充：

```json
{
  "scripts": {
    "format": "prettier --write .",
    "format:check": "prettier --check ."
  }
}
```

脚手架阶段已提供上述开发、构建、测试、检查和格式化命令，后续新增命令必须继续保持 npm 作为唯一包管理器入口。

## 开发启动方式

开发阶段前端通过 Vite 启动。

默认访问地址由 Vite 输出，一般为：

```text
http://localhost:5173
```

后端 Spring Boot 服务默认运行在：

```text
http://localhost:9090
```

前后端同时联调时，需要分别启动：

```text
前端：npm run dev
后端：Spring Boot 本地服务
```

后端服务的具体启动方式遵守后端项目规则。

## API 代理

前端开发阶段统一通过 Vite 代理访问后端。

前端代码只请求：

```text
/api/...
```

开发代理转发到：

```text
http://localhost:9090/api/...
```

这样做的原因：

- 避免前端代码写死后端域名。
- 避免开发阶段跨域问题。
- 保持本地和部署环境请求路径一致。
- 方便后续切换后端地址。

## 环境变量

环境变量文件建议使用：

```text
.env.example
.env
.env.development
.env.production
```

仓库只提交字段完整但不含私密值的 `.env.example`。真实 `.env` 和各环境配置文件只保留在本地，并由 `.gitignore` 排除。

Vite 暴露给前端代码的环境变量必须以 `VITE_` 开头。

建议变量：

```text
VITE_APP_TITLE=个人博客
VITE_API_BASE=/api
```

不允许放入：

- 数据库密码。
- JWT 密钥。
- 后端签名密钥。
- 生产服务器私密信息。
- 管理员账号密码。

前端环境变量会进入浏览器环境，不能当作安全密钥保存位置。

## 请求基础路径

请求基础路径由环境变量控制：

```text
VITE_API_BASE=/api
```

`utils/request.ts` 从环境变量读取基础路径。

页面和业务接口文件不直接读取后端真实地址。

## 本地联调要求

进行前后端联调前，需要确认：

- 后端服务已启动。
- 后端端口为 `9090` 或已同步调整代理配置。
- 前端 Vite 服务已启动。
- 数据库中已有必要测试数据。
- 后端接口能通过 Swagger 或接口工具访问。
- 前端请求路径以 `/api` 开头。
- 浏览器控制台没有跨域错误。
- Network 面板能看到请求进入后端。

## 代理和跨域

本地开发优先使用 Vite 代理解决跨域。

不优先要求后端开放全局 CORS。

只有当前端不经过 Vite 代理、或者未来部署需要浏览器跨域访问后端时，才讨论后端 CORS 配置。

## 构建输出

生产构建通过：

```text
npm run build
```

构建输出目录默认为：

```text
dist/
```

`dist` 是构建产物，不作为源码维护重点。

是否把 `dist` 部署到 Nginx、对象存储或后端静态目录，属于部署阶段设计，不在当前工程规格中展开。

## 预览构建结果

本地预览构建结果使用：

```text
npm run preview
```

预览只用于检查构建后的静态资源是否能正常访问，不替代真实部署验收。

## 质量检查命令

开发提交前建议执行：

```text
npm run typecheck
npm run lint
npm run test:run
npm run build
```

如果只是本地快速开发，可以先执行：

```text
npm run typecheck
npm run test:run
```

但正式交付前必须完整执行质量检查。

## 端口约定

默认端口：

```text
前端 Vite：5173
后端 Spring Boot：9090
```

如果端口冲突，可以临时调整前端端口，但需要确保代理仍然指向正确后端地址。

端口变化不应写死在页面代码中。

## 文件编码

前端项目统一使用 UTF-8。

源码、配置文件、Markdown 文档都使用 UTF-8。

中文文档和中文注释不应出现乱码。

## Git 忽略规则

前端项目应忽略：

```text
node_modules/
dist/
coverage/
.env.local
.env.*.local
```

锁文件需要提交，例如：

```text
package-lock.json
```

不要提交本地私有环境变量文件。

## 禁止事项

禁止以下做法：

- 在页面代码中写死 `http://localhost:9090`。
- 把后端密钥写进 `.env`。
- 把管理员账号密码写进前端配置。
- 多个地方各自维护 API 基础地址。
- 跳过类型检查直接构建。
- 把 `node_modules` 或 `dist` 当源码提交。
- 通过后端 CORS 掩盖前端代理配置错误。
