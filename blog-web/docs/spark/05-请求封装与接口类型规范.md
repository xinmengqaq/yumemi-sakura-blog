# 05-请求封装与接口类型规范

## 设计目标

请求封装是本前端工程的重点。

本项目前端由 AI 交付，但请求封装仍是工程重点。它决定前端如何连接后端、如何拿到数据、如何展示数据、如何处理接口错误。

因此所有接口调用必须有统一入口，不能在页面中零散请求。

请求封装需要解决：

- 后端基础地址如何配置。
- 请求头如何统一处理。
- Token 如何携带。
- 后端统一响应结构如何解析。
- 后端错误信息如何转换成前端可展示内容。
- 接口请求参数和响应数据如何用 TypeScript 表达。
- 前台公开接口和后台管理接口如何共享请求规则。

## 文件位置

请求基础封装放在：

```text
src/utils/request.ts
```

业务接口方法放在：

```text
src/api/
```

接口类型放在：

```text
src/types/
```

示例：

```text
src/utils/request.ts
src/api/auth.ts
src/api/profile.ts
src/types/api.ts
src/types/auth.ts
src/types/profile.ts
```

## 统一请求实例

项目只能有一个基础 Axios 实例。

该实例负责：

- 设置基础请求路径。
- 设置超时时间。
- 自动携带 Token。
- 处理后端统一响应结构。
- 处理常见错误。
- 把错误转换成统一前端错误对象。

页面组件和业务接口文件都不能临时创建新的 Axios 实例。

## 基础路径

开发阶段，前端请求统一从 `/api` 发出。

Vite 负责把 `/api` 代理到 Spring Boot 后端服务。

示例：

```text
前端请求：/api/admin/login
代理目标：http://localhost:9090/api/admin/login
```

前端代码中不直接写死 `http://localhost:9090`。

后端地址应通过环境变量或 Vite 代理配置管理。

## 后端统一响应结构

后端接口统一返回 `Result<T>` 结构。

前端应定义对应类型：

```ts
export type ApiResult<T> = {
  code: string
  msg: string
  data: T
}
```

业务代码主要关心 `data`，但不能忽略 `code` 和 `msg`。

请求封装应统一判断后端业务状态：

- 成功：返回 `data`。
- 失败：抛出包含 `code` 和 `msg` 的前端错误对象。

## 分页响应类型

后端分页接口应使用统一分页类型表示。

前端应定义：

```ts
export type PageResult<T> = {
  total: number
  records: T[]
}
```

如果后端实际分页字段发生变化，以后端接口文档和真实响应为准，同步调整类型。

## 请求方法命名

业务接口方法命名要表达业务动作。

推荐：

```ts
login()
logout()
getCurrentProfile()
updateProfile()
changePassword()
getArticleList()
getArticleDetail()
createArticle()
updateArticle()
deleteArticle()
```

禁止使用含义不清的命名，例如：

```ts
request1()
handleData()
getInfo()
submit()
```

## 请求参数类型

每个有请求体或查询参数的接口都应定义参数类型。

示例：

```ts
export type LoginParams = {
  username: string
  password: string
}
```

类型名称可以结合后端 DTO 含义，但不需要完全照搬 Java 类名。

例如后端是 `AdminLoginDTO`，前端可以命名为 `LoginParams`，但字段含义必须一致。

## 响应数据类型

每个接口返回的数据都应定义响应类型。

示例：

```ts
export type LoginUser = {
  id: number
  username: string
  name: string
  role: string
  avatar?: string
  token: string
}
```

响应类型应以真实后端响应为准。

不能为了页面展示方便，凭空增加后端没有返回的业务字段。

如果页面需要派生展示字段，应在页面或转换函数中生成，不应伪装成后端响应字段。

## Token 携带规范

登录成功后，前端保存后端返回的 Token。

后续需要登录的接口请求，由请求拦截器统一添加请求头：

```http
Authorization: Bearer <token>
```

页面和业务接口方法不手动拼接 `Authorization`。

Token 的读取应通过统一本地存储封装完成，不在多个文件里直接操作 `localStorage`。

## 401 处理规范

后端返回未登录、Token 无效或 Token 过期时，前端统一处理：

- 清理本地 Token。
- 清理当前登录用户基础信息。
- 跳转到后台登录页。
- 展示可理解的登录失效提示。

具体是否立即跳转，由请求封装和路由守卫配合决定。

页面不应各自重复写一套 401 处理逻辑。

## 错误对象规范

请求失败时，前端统一转换成错误对象。

建议字段：

```ts
export type ApiError = {
  code: string
  message: string
  status?: number
}
```

其中：

- `code` 来自后端业务错误码或前端自定义错误码。
- `message` 用于页面展示。
- `status` 表示 HTTP 状态码，可选。

页面展示错误时优先使用 `message`。

## 网络错误处理

如果请求没有到达后端，例如网络断开、后端未启动、代理配置错误，应给出和业务错误不同的提示。

例如：

```text
无法连接后端服务，请确认服务是否已启动
```

网络错误不能伪装成“用户名或密码错误”等业务错误。

## 接口文件职责

`src/api/*.ts` 文件只负责声明接口函数。

它可以：

- 引用 `utils/request.ts`。
- 引用 `types/*.ts`。
- 传递请求参数。
- 返回类型明确的 Promise。

它不应该：

- 操作页面跳转。
- 操作 DOM。
- 直接展示提示。
- 读写组件状态。
- 拼接完整后端域名。
- 创建新的 Axios 实例。

## 页面调用接口规范

页面调用接口时，应该处理四类状态：

- 初始状态。
- 加载中。
- 成功数据。
- 错误信息。

即使 UI 由 agent skills 生成，也必须把这四类状态接入页面。

页面不能只写成功态，忽略加载、空数据和错误。

## 空数据规范

列表接口返回空数组时，页面应展示空状态。

详情接口返回空数据或后端返回不存在时，页面应展示错误状态或跳转到对应错误页面。

空数据不是接口失败，不能和网络错误混为一类。

## 接口变更规则

后端接口字段发生变化时，前端必须同步修改：

- `types` 中的请求参数类型。
- `types` 中的响应数据类型。
- `api` 中的接口函数。
- 依赖该字段的页面展示。
- 对应测试。

不能只改页面，不改类型。

## 禁止事项

禁止以下做法：

- 在页面组件中直接写 `axios.get(...)`。
- 在页面组件中写死完整后端地址。
- 在多个文件里重复处理 Token 请求头。
- 在多个页面里重复判断 `code === '401'`。
- 用 `any` 跳过接口类型定义。
- 把后端没有返回的字段写进响应类型。
- 用假数据冒充真实后端接口。
- 在请求层处理具体页面展示细节。
