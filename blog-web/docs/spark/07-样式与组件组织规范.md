# 07-样式与组件组织规范

## 设计目标

本项目不要求用户手写复杂 UI。

样式和组件组织的主要目标是：

- 保证页面展示后端数据时清晰易读。
- 保证前台和后台视觉边界清楚。
- 保证页面代码能接入既定工程结构。
- 避免页面样式散乱、组件职责混乱。
- 不把项目做成死板的企业后台模板。

视觉主题应当舒服、方便、有活力，适合个人博客项目，而不是过度企业化、灰沉、密集和压抑。

UI 视觉、页面结构、布局细节、动效细节后续主要由项目级 `.agents` 目录中的 agent skills 约束和生成。本规格只定义前端工程中的样式和组件组织边界。

## 视觉方向

整体视觉应符合个人博客系统气质。

推荐方向：

- 清爽。
- 轻快。
- 有呼吸感。
- 内容层级明确。
- 操作入口清楚。
- 颜色有活力但不刺眼。
- 前台偏阅读体验。
- 后台偏管理效率，但不做成传统沉闷企业后台。

避免方向：

- 大面积深灰企业后台风。
- 过度密集的信息堆叠。
- 单调的蓝灰色系统模板感。
- 过度装饰导致数据难读。
- 为了好看牺牲表单、列表、按钮的可用性。
- 每个页面风格完全不同。

## 前台和后台视觉边界

前台和后台可以共享基础设计变量，但视觉侧重点不同。

前台页面：

- 更强调阅读体验和内容氛围。
- 留白可以更宽松。
- 图片、文章摘要、标签等内容展示可以更有生活感。
- 交互应轻量，不干扰阅读。

后台页面：

- 更强调信息扫描和操作效率。
- 表单、列表、筛选、状态展示要清晰。
- 不必做成企业系统的压抑风格。
- 可以保持轻快、有活力，但不能影响管理操作。

## 后台视口范围

后台页面只按桌面端管理工作台设计和验收，不做手机端、移动端或小屏专用适配。

后台布局应优先保证桌面宽度下的信息扫描、表单编辑和列表操作效率。真实浏览器验收使用桌面宽度，例如 1440 或 1536，不要求 390 等移动端宽度。

后台页面可以设置合理的桌面最小宽度，避免为了兼容移动端把表格、表单和侧边栏压缩到不可用。

## 图片与图标使用边界

图标用于按钮、菜单、状态提示、辅助说明等符号化场景。不是所有按钮、字段和区块都必须配图标，文字已经清楚时可以不用图标。

图片用于头像、封面、上传预览、文件图片预览、文章配图、相册内容等真实视觉对象。该用图片的地方必须使用图片或图片占位，不能用图标顶替。

图片加载失败或数据为空时，可以使用明确的图片占位、纯色默认头像或空状态文案；除非设计文档明确要求，不用图标替代头像、封面和预览图。

## 样式目录

全局样式放在：

```text
src/styles/
```

建议结构：

```text
src/styles/
├─ reset.css
├─ variables.css
└─ global.css
```

### reset.css

负责消除浏览器默认样式差异。

### variables.css

负责定义基础设计变量。

例如：

- 颜色。
- 字体。
- 间距。
- 圆角。
- 阴影。
- 层级。
- 过渡时间。

### global.css

负责项目全局基础样式。

例如：

- 页面背景。
- 字体渲染。
- 链接默认样式。
- 滚动条基础样式。
- 根节点尺寸。

## 样式编写方式

样式方案由 AI 根据页面复杂度和 agent skills 规则选择。

可以使用普通 CSS、CSS Modules、Tailwind CSS 或 UI 组件库配套样式方案。

无论使用哪种方案，都必须保证样式文件位置清楚、页面可维护、视觉方向统一。

## 设计变量

项目应尽量使用 CSS 变量保存基础视觉规则。

示例：

```css
:root {
  --color-primary: #4f7cff;
  --color-success: #22a06b;
  --color-warning: #f59e0b;
  --color-danger: #e5484d;
  --color-text: #1f2937;
  --color-text-muted: #6b7280;
  --color-bg: #f7f8fb;
  --radius-md: 8px;
  --space-md: 16px;
}
```

变量不是最终视觉方案，只是约束工程使用方式。

具体配色、布局和组件细节可以由 agent skills 根据项目阶段生成和调整。

## 组件目录

公共组件放在：

```text
src/components/
```

公共组件应服务复用，而不是为了拆文件而拆文件。

建议优先沉淀这些和数据展示相关的基础组件：

```text
src/components/
├─ LoadingState/
├─ EmptyState/
├─ ErrorState/
├─ PageHeader/
├─ DataSection/
└─ ConfirmDialog/
```

这些组件的重点不是炫技，而是让页面稳定表达：

- 正在加载。
- 暂无数据。
- 加载失败。
- 页面标题。
- 数据区块。
- 危险操作确认。

## 组件职责

公共组件应该满足：

- 通过 Props 接收数据。
- 不直接调用后端接口。
- 不直接读写 Token。
- 不依赖具体业务模块。
- 不伪造后端数据。
- 不把页面级状态藏在组件内部。

页面组件负责连接后端数据，公共组件负责展示传入的数据。

## 页面样式

页面样式可以放在页面同目录，也可以按组件目录组织。

推荐简单规则：

- 全局通用样式放 `src/styles`。
- 公共组件样式放对应组件目录。
- 页面专属样式放页面同目录或页面对应样式文件。
- 不同页面不要随意共享临时样式类名。

示例：

```text
src/views/admin/profile/
├─ ProfileView.tsx
└─ ProfileView.css
```

或：

```text
src/components/LoadingState/
├─ LoadingState.tsx
└─ LoadingState.css
```

## UI 组件库

允许 AI 根据页面复杂度选择外部 UI 组件库或局部组件依赖。

后台表格、表单、弹窗、消息提示、抽屉、菜单等交互密集场景，可以使用成熟 UI 组件库提高交付质量。

日期选择、上传、富文本、代码高亮、可视化、动效等复杂能力，也可以引入对应外部依赖，不要求第一版全部手写。

引入 UI 组件库前必须确认：

- 是否符合个人博客的视觉方向。
- 是否会让项目变成死板企业后台风。
- 是否能和现有路由、请求、类型、状态规范配合。
- 是否不会破坏请求、类型、状态和测试边界。

引入动画库时必须确认：

- 是否服务真实交互反馈或页面质感。
- 是否不会遮挡数据、影响表单操作或造成布局跳动。
- 是否能在真实浏览器验收中稳定运行。

## agent skills 协作

UI 视觉、页面结构、布局细节和动效细节，后续主要由项目级 `.agents` 目录中的 agent skills 约束和生成。

前端工程规格不重复定义 agent skills 已经负责的视觉规则。

当 `.agents` 目录移动到项目上级目录后，前端项目和后端项目都应遵守上级统一 agent 规则。

本规格只要求 agent skills 生成的页面代码接入工程边界：

- 页面放在 `views`。
- 接口请求放在 `api`。
- 类型定义放在 `types`。
- 全局状态放在 `store`。
- 通用请求实例放在 `utils/request.ts`。
- 公共组件放在 `components`。
- 全局样式放在 `styles`。

## 数据展示规范

页面展示后端数据时，需要明确区分：

- 加载中。
- 加载失败。
- 空数据。
- 正常数据。
- 操作成功。
- 操作失败。

视觉可以由 agent skills 设计，但这些状态必须在页面中真实存在。

不能只设计成功态页面。

## 禁止事项

禁止以下做法：

- 把样式全部写成无结构的大文件。
- 每个页面随意定义完全不同的视觉体系。
- 为了视觉效果写死后端数据。
- 公共组件直接请求后端接口。
- 公共组件直接读写登录 Token。
- 引入 UI 组件库后破坏项目视觉气质。
- 把后台做成死板沉闷的企业模板风。
- 用视觉装饰掩盖数据加载、错误和空状态缺失。
- 为后台管理界面补移动端、小屏端或手机端专用布局。
- 把应展示图片的头像、封面、上传预览、文件图片预览等场景改成图标。
