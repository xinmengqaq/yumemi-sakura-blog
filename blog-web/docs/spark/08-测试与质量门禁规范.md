# 08-测试与质量门禁规范

## 测试定位

本项目的前端测试重点是流转逻辑，不测试 UI 视觉细节。

这里的流转逻辑包括：

- 请求是否正确发出。
- Token 是否正确保存和携带。
- 登录态是否正确变化。
- 接口成功后数据是否进入页面。
- 接口失败后错误是否被识别。
- 401 是否触发清理登录态。
- 路由守卫是否按规则跳转。
- 表单提交是否调用正确接口。
- 页面是否区分加载中、空数据、错误和正常数据。

测试不负责验证：

- 颜色是否好看。
- 圆角是否合适。
- 间距是否舒服。
- 动效是否自然。
- 页面视觉是否高级。
- 浏览器截图是否符合审美预期。

真实浏览器验收、截图检查、视觉调整和交互体验验证，交给 AI 根据项目级 `.agents` 规则完成，并作为前端实现交付前的必要验收。后台管理界面只做桌面端验收，不做移动端或手机端验收。

## 测试工具

当前阶段使用：

```text
Vitest
Testing Library
```

Vitest 用于运行测试。

Testing Library 用于从用户行为角度验证组件和页面流转。

单元测试默认不使用复杂 E2E 测试框架。

真实浏览器验收可以由 AI 使用 Playwright 或同等浏览器自动化工具完成。

## 测试文件位置

测试文件可以与被测试文件放在同目录。

示例：

```text
src/utils/request.ts
src/utils/request.test.ts

src/store/auth.ts
src/store/auth.test.ts
```

页面级测试也可以放在页面同目录：

```text
src/views/admin/login/LoginView.tsx
src/views/admin/login/LoginView.test.tsx
```

测试辅助工具放在：

```text
src/test/
```

示例：

```text
src/test/
├─ setup.ts
├─ render.tsx
└─ mocks/
```

## 优先测试对象

优先测试和前后端连接强相关的代码。

第一优先级：

- `utils/request.ts`
- `utils/storage.ts`
- `store/auth.ts`
- `router/guards.tsx`
- `api/*.ts`

第二优先级：

- 登录页流转。
- 资料页数据加载流转。
- 列表页加载、空数据、错误流转。
- 表单提交成功和失败流转。

第三优先级：

- 纯展示组件的状态切换。
- 通用工具函数。

纯视觉组件没有复杂逻辑时，不强制写测试。

## 请求封装测试

请求封装必须覆盖：

- 有 Token 时自动添加 `Authorization` 请求头。
- 无 Token 时不添加认证请求头。
- 后端成功响应时返回 `data`。
- 后端业务失败时抛出统一错误对象。
- HTTP 401 或后端未登录结果会清理登录态。
- 网络错误能转换成可理解错误信息。

测试目标是验证请求流转，不验证请求按钮长什么样。

## 登录态测试

登录态测试必须覆盖：

- 登录成功后保存 Token。
- 登录成功后保存当前用户基础信息。
- 退出登录后清理 Token。
- 退出登录后清理当前用户基础信息。
- 页面刷新后可以从本地缓存恢复登录态。
- Token 失效后能清理登录态。

不测试登录页视觉布局。

## 路由守卫测试

路由守卫测试必须覆盖：

- 未登录访问后台页面时跳转登录页。
- 已登录访问后台登录页时跳转后台首页。
- 公开前台页面不要求登录。
- 未匹配路径进入 404。
- Token 被判定无效时清理登录态。

测试只验证路由结果，不验证跳转动画和页面样式。

## 页面流转测试

页面流转测试重点验证页面如何响应数据状态。

至少覆盖：

- 初次进入页面显示加载状态。
- 接口成功后显示后端返回的数据。
- 接口返回空数组时显示空状态。
- 接口失败时显示错误状态。
- 用户操作后调用正确接口。
- 操作成功后页面状态按规则更新。

不测试颜色、间距、卡片样式、图标是否美观，也不为了方便测试把应使用图片的头像、封面、上传预览等场景改成图标。

## 表单流转测试

表单测试重点验证数据提交过程。

至少覆盖：

- 用户输入内容后表单状态更新。
- 点击提交时调用对应接口。
- 提交中避免重复提交。
- 提交成功后展示成功状态或执行跳转。
- 提交失败后展示后端错误信息。
- 必填字段缺失时不发出无效请求。

不测试输入框宽度、按钮颜色、提示动画。

## Mock 规范

前端测试可以 mock 后端响应，但 mock 数据必须遵守接口类型。

mock 数据不能随意添加后端不存在的字段。

推荐把测试 mock 放在：

```text
src/test/mocks/
```

mock 数据应围绕行为最小化，不为了视觉丰满伪造大量字段。

## 不测试 UI 的边界

“不测试 UI”不是“不验证页面”。

测试仍然要验证：

- 数据是否出现在页面上。
- 错误提示是否出现。
- 空状态是否出现。
- 按钮点击是否触发流程。
- 表单输入是否影响提交参数。

不验证：

- 具体颜色。
- 字体大小。
- 阴影。
- 圆角。
- 复杂布局。
- 动画。
- 截图像不像设计稿。

## 真实浏览器验收

真实浏览器验收不作为单元测试职责，但属于 AI 前端交付职责。

AI 完成前端页面后，必须使用真实浏览器或浏览器自动化验证核心流程，重点看：

- 页面是否能正常打开。
- 数据是否真实从后端接口进入页面。
- 页面是否没有明显遮挡、错位、文字溢出。
- 表单和按钮是否能实际操作。
- 登录、跳转、退出等流程是否可完成。

这部分验收属于实现阶段的 AI 工作，不写入单元测试范围，但交付说明中必须报告验收结果。

## 质量门禁

前端交付前至少应通过：

```text
npm run typecheck
npm run lint
npm run test
npm run build
```

如果某一项失败，不能认为当前前端工程可交付。

如果后续增加格式化检查，可以补充：

```text
npm run format:check
```

## 覆盖率态度

当前阶段不设置硬性覆盖率数字。

覆盖率不能替代关键流转验证。

一个只测 UI 快照、不测请求和登录态的高覆盖率项目，没有意义。

优先保证关键路径有测试：

- 请求封装。
- 登录态。
- 路由守卫。
- 接口成功和失败。
- 页面数据流转。

## 禁止事项

禁止以下做法：

- 写快照测试代替行为测试。
- 为了覆盖率测试无意义实现细节。
- 测试颜色、圆角、阴影等视觉细节。
- mock 数据字段和后端类型不一致。
- 只测试成功态，不测试失败态。
- 登录态测试只保存 Token，不验证清理流程。
- 页面测试绕过真实调用链，直接塞最终展示数据。
