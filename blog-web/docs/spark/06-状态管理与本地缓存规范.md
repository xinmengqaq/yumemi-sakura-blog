# 06-状态管理与本地缓存规范

## 设计目标

状态管理用于保存前端运行过程中需要跨组件、跨页面使用的数据。

本项目前端主要由 AI 交付，状态管理要保证服务端数据、客户端状态和本地缓存边界清晰。

状态管理要解决：

- 登录 Token 保存在哪里。
- 当前登录用户基础信息如何共享。
- 页面刷新后登录态如何恢复。
- 接口请求中的加载、成功、失败状态如何表达。
- 哪些数据应该存在全局状态，哪些数据应该只留在页面内部。
- 本地缓存如何统一读写。

## 状态管理方案

客户端全局状态优先使用 Zustand。

服务端状态优先使用 TanStack Query。

本地持久化缓存通过 `storage.ts` 统一封装。

React 自带状态仍用于页面内部局部状态，例如表单输入、弹窗开关、临时选择项。

## store 目录职责

全局状态放在：

```text
src/store/
```

可以规划：

```text
src/store/
├─ auth.ts
└─ layout.ts
```

### auth.ts

负责登录态相关客户端状态。

包括：

- Token。
- 当前登录用户基础信息。
- 是否已登录。
- 登录态初始化。
- 清理登录态。

`auth.ts` 不负责请求后端接口。登录、退出、校验 Token 等请求放在 `api` 目录，服务端请求流转放在 `queries` 目录。

### layout.ts

负责前端布局状态。

例如：

- 后台侧边栏是否折叠。
- 当前主题模式。
- 后台导航分组是否展开。

布局状态不能混入业务数据。

后台管理界面不做移动端适配，因此不保存移动端菜单、移动端抽屉或小屏专用布局状态。

## 本地缓存封装

本地缓存基础封装放在：

```text
src/utils/storage.ts
```

项目中禁止在多个文件里直接调用 `localStorage`。

所有本地缓存读写必须经过 `storage.ts`。

这样做的原因：

- 统一 key 命名。
- 方便处理 JSON 序列化。
- 方便后续调整存储策略。
- 避免多个页面重复写错 key。
- 方便测试。

## 本地缓存范围

允许缓存：

- 登录 Token。
- 当前登录用户基础信息。
- 用户界面偏好，例如侧边栏折叠状态。
- 不敏感的前端展示设置。

不允许缓存：

- 明文密码。
- 数据库连接信息。
- 后端密钥。
- JWT 签名密钥。
- 未脱敏的敏感数据。
- 临时表单中的敏感输入。

## Token 保存策略

Token 使用 `localStorage` 保存。

原因：

- 实现直观。
- 页面刷新后仍可恢复登录态。
- 方便观察前端如何携带 Token 请求后端。
- 和当前后端 JWT 无状态方案匹配。

后续如果需要更高安全要求，再评估 httpOnly Cookie 等方案。

## Token 使用规则

Token 只在以下位置被直接读取：

- `src/store/auth.ts`
- `src/utils/request.ts`
- `src/utils/storage.ts`

其他页面或组件不直接读取 Token。

页面只关心是否已登录和当前用户信息，不关心 Token 字符串本身。

## 登录态初始化

应用启动时，需要从本地缓存读取 Token 和用户基础信息。

如果存在 Token：

- 先恢复前端登录态。
- 访问需要登录的页面时，可以调用后端校验接口确认 Token 是否有效。
- 如果后端确认 Token 无效，清理本地登录态并跳转登录页。

如果不存在 Token：

- 视为未登录。
- 访问后台受保护页面时跳转登录页。

## 当前用户信息

当前用户基础信息可以缓存在本地，用于页面刷新后的临时展示。

但后端仍然是最终可信来源。

进入后台后，应通过后端接口获取当前用户最新信息，避免本地缓存过期。

## 接口请求状态

每个页面读取后端数据时，至少要能表达四类状态：

- `idle`：尚未请求。
- `loading`：请求中。
- `success`：请求成功。
- `error`：请求失败。

也可以使用更简单的组合：

```ts
{
  loading: boolean
  data: T | null
  error: string | null
}
```

但不能只处理成功数据，忽略加载中和错误状态。

## 页面局部状态

只服务当前页面的状态，应放在页面组件内部。

例如：

- 表单输入值。
- 当前打开的弹窗。
- 当前选中的标签。
- 当前页码。
- 当前搜索关键词。

这些状态不应放入全局 `store`，除非多个页面确实需要共享。

## 全局状态

适合放入全局状态的数据包括：

- 登录态。
- 当前用户基础信息。
- 全局布局偏好。
- 全局主题偏好。
- 跨页面共享但不适合从 URL 表达的数据。

后端业务列表数据默认不放入全局状态。

例如文章列表、分类列表、评论列表等，应优先由对应页面请求后端获取。

## URL 状态

可分享、可刷新恢复的页面状态，应优先放在 URL 查询参数中。

例如：

- 页码。
- 搜索关键词。
- 状态筛选。
- 分类筛选。
- 标签筛选。

这样刷新页面或复制链接时，页面状态不会丢失。

## 表单状态

表单输入状态优先保存在页面内部。

表单提交后，调用 `api` 中的接口方法。

提交成功后，根据业务需要刷新数据、跳转页面或展示成功提示。

表单状态不写入本地缓存，除非后续明确需要草稿能力。

## 缓存失效规则

登录失效时必须清理：

- Token。
- 当前登录用户基础信息。
- 和登录用户强相关的本地缓存。

退出登录时也必须执行同样清理。

不能只删除 Token，留下用户信息造成页面误判。

## 多端登录与旧 Token

后端通过密码版本控制旧 Token 失效。

前端不需要解析 JWT，也不需要判断密码版本。

前端只负责：

- 带 Token 请求后端。
- 接收后端返回的未登录或过期结果。
- 清理本地登录态。
- 引导用户重新登录。

## 禁止事项

禁止以下做法：

- 多个页面直接读写 `localStorage`。
- 把密码保存到本地缓存。
- 前端自行解析 JWT 判断是否过期。
- 把业务列表数据全部塞进全局状态。
- 为了方便把所有页面状态都放进 `store`。
- 登录失效后只清 Token 不清用户信息。
- 用全局状态替代 URL 查询参数保存分页和筛选条件。
