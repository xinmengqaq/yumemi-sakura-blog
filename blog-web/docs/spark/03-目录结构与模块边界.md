# 03-目录结构与模块边界

## 目录设计目标

前端目录结构要符合国内后台和博客前端项目的常见习惯，同时保持清晰的维护边界。

本项目的前端主要由 AI 交付，目录结构要优先保证边界清楚、可维护、可审查，避免生成代码时把请求、状态、页面和样式混在一起。

目录结构需要满足三个目标：

- 人能快速判断代码应该放在哪里。
- 生成代码时不能把请求、类型、状态、页面和样式混在一起。
- 后续前台和后台都能放在同一个前端工程中继续扩展。

## 推荐目录结构

```text
src/
├─ api/
├─ assets/
├─ components/
├─ hooks/
├─ layout/
├─ queries/
├─ router/
├─ store/
├─ styles/
├─ types/
├─ utils/
└─ views/
```

## api

`api` 目录放后端接口请求方法。

所有后端请求都必须先封装在 `api` 目录中，页面不能直接调用 Axios，也不能在页面中拼接完整后端地址。

接口文件按业务模块拆分。

示例：

```text
src/api/
├─ auth.ts
├─ profile.ts
├─ article.ts
├─ category.ts
└─ siteConfig.ts
```

当前阶段只创建已经需要联调的接口文件，不提前创建所有业务文件。

脚手架阶段只落地认证、请求、路由、布局、状态、样式、测试辅助等工程基础目录和文件，不提前创建文章、分类、标签等后端尚未进入联调的业务模块文件。

`api` 文件只负责请求后端，不负责页面状态，不负责样式，不负责复杂展示逻辑。

## queries

`queries` 目录放 TanStack Query 相关封装。

它负责把 `api` 中的接口方法组织成页面可直接使用的数据查询和变更逻辑。

示例：

```text
src/queries/
├─ auth.ts
├─ profile.ts
├─ article.ts
└─ category.ts
```

`queries` 可以处理：

- 查询 key。
- 加载状态。
- 错误状态。
- 缓存失效。
- 重新请求。
- 新增、修改、删除后的列表刷新。

页面优先通过 `queries` 获取后端数据，不在页面里直接管理复杂请求状态。

## assets

`assets` 目录放前端项目自带的静态资源。

主要包括：

- 图片。
- 图标。
- 字体文件。
- 本地静态媒体资源。

如果图片或文件来自后端接口返回的 URL，则不放在 `assets`，而是作为后端数据展示。

图片和图标不能混用：头像、封面、上传预览、文件图片预览、文章配图、相册内容等场景应使用图片或图片占位；按钮、菜单、状态提示等符号化场景才使用图标。该用图片的地方不要用图标顶替。

## components

`components` 目录放公共展示组件。

公共组件可以被前台和后台复用，但不能直接绑定具体业务接口。

示例：

```text
src/components/
├─ LoadingState/
├─ EmptyState/
├─ ErrorState/
├─ PageHeader/
└─ ConfirmDialog/
```

公共组件应该满足：

- 组件数据通过 Props 传入。
- 不在组件内部直接请求后端。
- 不在组件内部读写 Token。
- 不在组件内部伪造后端数据。
- 不把复杂业务逻辑藏在展示组件里。

## hooks

`hooks` 目录放通用 Hook。

通用 Hook 应该和具体业务弱绑定，例如：

- 本地缓存读取。
- 防抖。
- 页面标题。
- 简单异步状态。
- 浏览器尺寸监听。

如果某个 Hook 只服务一个业务页面，并且和具体接口强相关，可以先放在对应页面附近；当多个页面复用时，再提升到 `hooks`。

## layout

`layout` 目录放页面布局。

前台和后台可以使用不同布局。

示例：

```text
src/layout/
├─ FrontLayout.tsx
├─ AdminLayout.tsx
└─ BlankLayout.tsx
```

布局只负责页面框架，例如头部、侧边栏、内容区、底部，不负责具体业务请求。

## router

`router` 目录放路由配置和路由守卫。

示例：

```text
src/router/
├─ index.tsx
├─ routes.tsx
└─ guards.tsx
```

路由配置负责定义访问路径、页面组件和布局关系。

路由守卫负责处理是否需要登录、未登录跳转、无效路径跳转等通用规则。

路由文件不写具体业务接口请求，不处理页面表单逻辑。

## store

`store` 目录放全局状态。

`store` 目录放客户端全局状态，优先使用 Zustand。

`store` 只保存少量跨页面客户端状态，例如登录态、用户基础信息、前端布局状态。

示例：

```text
src/store/
├─ auth.ts
└─ layout.ts
```

`store` 不保存所有后端接口数据。后端列表、详情、分页、筛选结果等服务端状态优先交给 TanStack Query。

## styles

`styles` 目录放全局样式。

示例：

```text
src/styles/
├─ reset.css
├─ variables.css
└─ global.css
```

## types

`types` 目录放 TypeScript 类型定义。

通用类型和接口类型都放在这里，并按模块拆分。

示例：

```text
src/types/
├─ api.ts
├─ auth.ts
├─ profile.ts
├─ article.ts
└─ page.ts
```

`api.ts` 放统一响应结构，例如后端的 `Result<T>` 和分页结构。

业务模块类型要和后端 DTO、VO 含义对应，不能随意改业务语义。

页面展示需要什么字段，应优先从后端响应类型出发，而不是前端自己凭空造字段。

## utils

`utils` 目录放工具函数和工程基础封装。

示例：

```text
src/utils/
├─ request.ts
├─ storage.ts
├─ env.ts
├─ error.ts
└─ format.ts
```

其中 `request.ts` 负责创建统一 Axios 实例，并处理 Token、响应结构和错误转换。

`utils` 不应该依赖具体页面，也不应该依赖具体业务模块。

## views

`views` 目录放路由页面。

前台页面和后台页面在 `views` 下分目录保存。

示例：

```text
src/views/
├─ front/
├─ admin/
└─ error/
```

这里的 `admin` 只表示后台管理区域，不代表整个前端工程只服务后台。

页面组件负责把接口数据展示出来，处理用户操作入口，并组合公共组件、布局和 API 方法。

页面中禁止：

- 直接创建 Axios 实例。
- 直接拼接完整后端地址。
- 写死后端返回数据。
- 大量使用 `any`。
- 把请求、类型、状态、样式全部堆在一个大文件里。
- 为了展示效果伪造后端不存在的业务字段。

## 模块命名规则

文件命名应清晰表达职责。

推荐：

- 接口文件使用业务名，例如 `auth.ts`、`article.ts`。
- 类型文件使用业务名，例如 `auth.ts`、`article.ts`。
- 页面文件使用 PascalCase，例如 `LoginView.tsx`、`ArticleListView.tsx`。
- 组件目录和组件文件使用 PascalCase。
- 工具文件使用 camelCase，例如 `request.ts`、`storage.ts`。

## 引用边界规则

引用关系遵循以下规则：

- `views` 可以引用 `api`、`queries`、`types`、`components`、`hooks`、`store`、`utils`。
- `views` 优先通过 `queries` 获取后端数据，避免直接管理复杂请求状态。
- `queries` 可以引用 `api` 和 `types`。
- `api` 可以引用 `utils/request` 和 `types`。
- `components` 可以引用 `types`、`styles`、`utils`，但不直接引用 `api`。
- `router` 可以引用 `views`、`layout`、`store`。
- `store` 可以引用 `api`、`types`、`utils`。
- `utils` 不引用 `views`、`router`、`layout`。
- `types` 不引用业务实现代码。

不能出现通用层反向依赖页面层。

## 新增模块规则

新增前端模块前必须先确认后端模块是否已经进入开发阶段。

如果后端模块还没有对应接口，前端只允许预留路由或占位页面，不允许编写完整假数据业务页面。

新增模块时，优先补齐：

```text
src/types/<module>.ts
src/api/<module>.ts
src/queries/<module>.ts
src/views/<area>/<ModuleView>.tsx
```

其中 `<area>` 根据前台或后台路由区域决定。

如果模块需要跨页面共享客户端状态，再补充 `src/store/<module>.ts`。
