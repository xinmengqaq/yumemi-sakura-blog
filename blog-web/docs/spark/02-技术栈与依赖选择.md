# 02-技术栈与依赖选择

## 技术栈定位

前端工程使用 React + TypeScript + Vite 作为核心技术栈。

本项目的前端主要由 AI 按规格交付，用户不需要手写复杂 UI 组件，也不需要从零实现按钮、表格、弹窗、表单控件等视觉组件。

本项目的技术栈选择重点是：

- 支撑 AI 稳定交付前台和后台页面。
- 保证前端能可靠调用后端接口。
- 保证接口类型、登录态、请求缓存和错误处理有清晰边界。
- 保证后端数据能被正确展示到页面。
- 保证加载中、空数据、错误提示和未登录跳转等状态完整。
- 保证测试能验证前后端连接相关行为。

视觉设计、复杂组件结构、样式细节和交互动效可以由 agent skills 辅助生成，但生成结果必须符合本项目工程规范，不能破坏目录结构、数据流、类型约束和请求封装规则。

## 核心依赖

### React

React 用于构建前端界面。

本项目使用函数组件和 Hook，不使用 Class Component。

React 的工程重点放在数据如何进入组件、组件如何根据数据渲染、用户操作如何触发接口请求，而不是手写底层 UI 控件。

### TypeScript

TypeScript 用于给前端代码增加类型约束。

接口响应、请求参数、组件 Props、状态数据都应尽量定义明确类型。

本项目尤其重视接口类型，因为前端展示的数据来自后端。页面需要清楚知道后端返回了什么字段、字段可能为空还是必填、请求参数需要什么结构。

禁止在业务代码中随意使用 `any`。如果暂时无法确定类型，应先定义最小可用类型，再随着接口和业务明确逐步补齐。

### Vite

Vite 用于前端开发服务器、构建和环境变量管理。

开发阶段通过 Vite 启动本地前端服务，并通过代理把 API 请求转发到 Spring Boot 后端。

Vite 的工程重点是开发服务、环境变量、代理配置和构建命令。

### React Router

React Router 用于前端路由管理。

前台页面和后台页面应通过路由结构区分，但不要求在当前阶段展开具体业务路由。

React Router 的工程重点是页面如何切换、哪些页面需要登录态、未登录时如何跳转，而不是复杂路由动画或花哨导航效果。

### Axios

Axios 用于封装 HTTP 请求。

所有接口请求必须通过统一 Axios 实例发出，不能在页面组件里直接零散调用 `fetch` 或创建多个临时请求实例。

Axios 是本项目的重点依赖。请求封装需要处理基础路径、请求头、Token、统一响应结构和错误信息。

### TanStack Query

TanStack Query 用于管理服务端状态。

它负责接口数据的加载状态、错误状态、缓存、重新请求和列表刷新。AI 交付页面时，应优先使用 TanStack Query 承接后端数据流，避免把请求状态散落在页面组件中。

Axios 负责“怎么请求”，TanStack Query 负责“请求结果在页面中如何流转”。

### Zustand

Zustand 用于管理少量跨页面客户端状态。

它适合保存登录态、当前用户基础信息、布局偏好、主题偏好等前端状态。

Zustand 不用于保存所有后端列表数据。文章列表、分类列表、评论列表等服务端数据优先交给 TanStack Query 管理。

### Vitest

Vitest 用于前端单元测试和基础行为测试。

测试重点放在请求封装、接口类型辅助函数、路由守卫、登录态读写和错误处理，不追求测试视觉样式。

### Testing Library

Testing Library 用于测试 React 组件的用户行为。

组件测试重点验证数据展示和用户操作结果，例如接口数据是否显示到页面、点击按钮是否触发请求、接口失败是否显示错误信息。

不要求测试颜色、圆角、布局细节、动画效果等纯视觉表现。

## 开发辅助依赖

### ESLint

ESLint 用于检查代码质量和常见错误。

它负责发现未使用变量、错误 Hook 用法、不稳定代码结构等问题。

### Prettier

Prettier 用于统一代码格式。

格式化规则不应该和业务逻辑混在一起讨论，代码风格交给工具自动处理。

## 可引入的交付依赖

### UI 组件库

允许 AI 根据页面复杂度引入外部成熟 UI 组件库或局部组件依赖。

UI 组件库的选择必须符合个人博客系统气质：舒服、方便、有活力，不能把后台做成死板沉闷的企业模板。

引入 UI 组件库后，仍然必须遵守本工程的数据流边界：接口请求放在 `api`，服务端状态放在 TanStack Query，客户端状态放在 Zustand，类型定义放在 `types`。

表格、表单、弹窗、抽屉、日期选择、上传、富文本、代码高亮、可视化等场景，如果外部组件库能明显提高质量和交付效率，可以直接引入，不要求第一版全部手写。

### 图标、图片和动画资源

允许引入图标库、图片资源和动画库提升页面质感。

图标只用于按钮、菜单、状态提示、空状态辅助符号等适合符号化表达的场景，不要求所有按钮、字段或区块都配图标。文案已经清楚时，宁可不用图标。

需要展示头像、封面、上传预览、文件图片预览、文章配图、相册内容等真实视觉对象时，应使用图片或图片占位，不能用图标替代图片。图片来自后端 URL 时按后端数据展示；项目自带图片资源才放入 `assets`。

图标、图片和动画不能影响数据可读性、表单可用性和核心操作效率。

动画库可以按需要引入，例如 GSAP 或其他适合 React 的动效依赖。动画不作为单元测试重点，真实浏览器验收时检查是否提升体验、是否影响页面使用、是否造成遮挡或布局跳动。

## 不优先引入的依赖

### Redux

当前工程优先使用 Zustand 管理客户端状态，不优先引入 Redux。

如果后续出现复杂状态审计、严格状态流或大型团队协作需求，再重新评估 Redux。

## 依赖引入规则

新增依赖前必须先判断：

- 当前问题是否可以用 React、TypeScript、浏览器 API 或已有依赖解决。
- 新依赖是否和“前后端连接、数据展示、状态处理、AI 交付质量”相关。
- 新依赖是否会成为长期工程基础。
- 新依赖是否会增加明显维护负担。
- 新依赖是否和当前项目阶段匹配。
- 新依赖是否需要写入对应工程规格文档。

不能为了“看起来专业”提前引入项目暂时用不到的库。

## 版本管理原则

依赖版本应由包管理器锁文件固定。

新增依赖后，需要同步提交依赖清单和锁文件，避免不同机器安装出不一致版本。

如果依赖存在大版本升级，升级前应确认破坏性变更，不直接盲目升级。

## 包管理器

前端项目建议使用 npm 作为包管理器。

原因是 npm 随 Node.js 默认安装，使用门槛低，命令资料多，适合当前项目。

如果后续需要更快安装速度或更严格的 monorepo 管理，再评估是否切换 pnpm。
