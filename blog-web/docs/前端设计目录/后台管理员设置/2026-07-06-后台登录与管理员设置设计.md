# 后台登录与管理员设置 UI 设计规格

## 设计目标

本规格用于设计个人博客后台的第一组真实后台页面：后台登录页、后台首页、后台框架和管理员设置页。

本阶段只接入后端第一阶段和第二阶段已经实现的能力，不设计文章、分类、标签、评论、统计等后续模块页面。

页面目标不是做展示型官网，而是做可验证、可复用、可继续扩展的后台管理界面。第一版必须能真实验证管理员登录、登录态保护、管理员资料维护、密码修改、Token 校验、Token 刷新和退出登录。

## 后端依据

本规格以 `D:/daimai/项目学习/springboot/` 中真实后端代码为准。

已确认的后端接口如下：

| 能力               | 方法  | 地址                          | 说明                                             |
| ------------------ | ----- | ----------------------------- | ------------------------------------------------ |
| 管理员登录         | POST  | `/api/admin/login`            | 不需要 Token，成功返回带 Token 的管理员信息      |
| 退出登录           | POST  | `/api/admin/logout`           | 需要 Token，后端无服务端会话，前端负责清理登录态 |
| 获取当前管理员资料 | GET   | `/api/admin/profile`          | 需要 Token，返回不带密码的管理员信息             |
| 修改管理员资料     | PUT   | `/api/admin/profile`          | 需要 Token，修改 `username`、`name` 和 `avatar`  |
| 修改管理员密码     | PATCH | `/api/admin/profile/password` | 需要 Token，旧密码校验成功后修改密码             |
| 校验 Token         | GET   | `/api/admin/validate`         | 需要 Token，成功返回 `{ valid: true }`           |
| 刷新 Token         | POST  | `/api/admin/refresh`          | 需要 Token，成功返回新 Token                     |

统一响应结构为：

```json
{
  "code": "200",
  "msg": "请求成功",
  "data": {}
}
```

登录接口返回的 `data` 是平铺的 `AdminVO`，不是 `token + user` 嵌套结构：

```ts
type AdminVO = {
  id: number
  username: string
  name: string
  role: string
  avatar?: string
  token?: string
}
```

前端当前 `LoginResult` 类型需要按真实后端契约修正。登录成功后，前端应从同一个 `AdminVO` 中拆出：

```text
token：data.token
currentUser：id / username / name / role / avatar
```

资料接口返回的 `AdminVO` 不包含 `token`。前端不能假设资料接口会返回新 Token。

## 页面范围

本阶段设计以下页面：

| 页面         | 路由                    | 说明                                |
| ------------ | ----------------------- | ----------------------------------- |
| 后台登录页   | `/admin/login`          | 管理员输入用户名和密码登录          |
| 后台首页     | `/admin`                | 登录后的默认落点，不做假统计        |
| 管理员设置页 | `/admin/settings/admin` | 验证资料维护、密码修改和 Token 能力 |

后台首页只展示真实登录态摘要和进入管理员设置的入口，不伪造文章数、访问量、评论数等后端尚未提供的数据。

## 视觉主题

主题名称：Build Studio 极简后台。

这个主题用于后台管理区，不用于前台博客展示区。

视觉目标：

- 安静。
- 简洁。
- 自然。
- 低存在感。
- 表单和状态优先。
- 通过排版、留白和细线建立层级，不通过卡片、阴影和装饰图标堆叠层级。

视觉规则：

| 项     | 规则                                                               |
| ------ | ------------------------------------------------------------------ |
| 底色   | 冷中性近白画布配白色内容区，不使用渐变、纹理或装饰图片             |
| 文字   | 深灰正文，中灰辅助文字，标题字重不超过 650                         |
| 线条   | 使用 1px 中性灰线分隔导航、章节和数据行                            |
| 强调色 | 单一低饱和赤陶红，只用于焦点、当前菜单和文本链接                   |
| 主操作 | 使用深灰实体按钮，不把强调色用于所有主要按钮                       |
| 危险色 | 深红色，仅用于退出、删除和危险确认                                 |
| 圆角   | 按钮和输入框统一 4px，后台操作面板统一 6px，弹窗最多 4px           |
| 阴影   | 页面、表单区和按钮无阴影，只有弹窗允许轻阴影                       |
| 图标   | 只保留真正帮助识别的菜单和操作图标，不给章节标题和提示框机械配图标 |
| 图片   | 头像、上传预览、文件图片预览等真实视觉对象必须使用图片或图片占位   |
| 字体   | 使用项目内置的 `Noto Sans SC Variable`，不依赖访问外部字体服务     |
| 装饰   | 禁止叶子方块标志、胶囊容器、彩色状态大块、无意义插图和假数据       |

视觉优化方向：

- 后台框架采用安静的近白工作台底色，侧边栏、顶部栏和内容区只用边界和留白区分层级。
- 桌面后台框架固定在视口内，侧边栏不参与内容滚动；只有右侧内容区滚动，滚动或拖动内容时侧边栏不产生位移。
- 顶部栏始终固定显示，不根据内容滚动隐藏，不使用透明模糊效果。
- 登录页使用左右分区布局：左侧承载纯文字品牌，右侧承载居中表单；不使用悬浮卡片或宣传页背景。
- 品牌只显示纯文字「个人博客后台」，不使用叶子图标或彩色品牌方块。
- 主按钮使用深灰实体按钮；语义状态色只用于成功、警告、错误等明确状态。
- 头像预览和顶部栏头像统一为图片容器，头像为空、`null`、空字符串或加载失败时显示纯灰色圆形占位。
- 动效只保留按钮按压、导航悬停和焦点变化，不引入自动动画或循环动画。

不采用以下风格：

- 不采用暗色发光 Bento 风格，因为它更像技术产品宣传页。
- 不采用暖色出版物风格或大面积米色，因为它会削弱后台表单的清晰度。
- 不采用传统沉闷企业后台模板，因为本项目仍是个人博客系统。
- 不采用蓝色卡片、图标方框、胶囊状态、重复说明文案等 AI 生成后台常见模式。

## 真实浏览器布局复核

以用户正常桌面浏览器工作区为准检查。浏览器自动化不得传入固定 viewport，不再为了截图机械设置为 `1440x900`；验收时记录浏览器实际可视区域即可。

首轮真实浏览器检查发现：

- 登录表单依赖任意偏左定位，宽屏下形成大面积无意义空白。
- 后台内容被限制在居中的 `880px` 窄栏，和侧边栏、顶部栏之间缺少稳定的版面关系。
- 管理员设置把章节标题与字段硬拆为左右两列，导致标题区过窄、字段区拥挤。
- 顶部栏左侧完全空置，页面上下文只能依赖内容区标题，视觉重心偏向右上角用户区域。

修正规则：

- 登录页采用约 `44% / 56%` 的左右分区，表单宽度稳定在 `400px` 左右并在右侧区域内居中。
- 后台侧边栏宽度约 `236px`，顶部栏高度约 `60px`，顶部栏左侧显示当前页面上下文。
- 后台内容区在正常宽屏浏览器下左对齐并随窗口扩展，最大宽度约 `1520px`，不再使用居中窄栏或在右侧留下大片空白。
- 后台首页使用横向管理员概览，把头像、身份字段和设置入口组织在同一个稳定网格内。
- 管理员设置的每个功能区使用完整宽度：标题在上、内容在下；资料、密码、登录状态分别形成独立白色操作面板。
- 操作面板只使用 1px 边框和 6px 圆角，不使用阴影，不在面板内继续嵌套卡片。

## 视口范围

本设计只面向桌面端后台管理工作台，不做手机端、移动端或小屏专用适配。

验收以用户正常桌面浏览器窗口为主，不以自动化工具预设的截图尺寸作为设计基准。后台布局可以保持合理的桌面最小宽度，不为移动端切换侧边栏、抽屉菜单或单独导航形态。

## 后台导航结构

后台侧边栏第一阶段只包含两个入口：

| 菜单       | 路由                    | 说明                          |
| ---------- | ----------------------- | ----------------------------- |
| 后台首页   | `/admin`                | 登录后的默认页面              |
| 管理员设置 | `/admin/settings/admin` | 管理员资料、密码和 Token 状态 |

当前菜单项需要有明确 active 状态。active 状态只能改变颜色、边框或背景，不能导致菜单项尺寸变化。

## 页面交互

### 登录

流程：

```text
输入用户名、密码
→ 点击「登录后台」
→ 调 POST /api/admin/login
→ 成功：保存 token 和管理员信息，进入 /admin
→ 失败：显示后端 msg
```

状态规则：

| 状态           | 行为                                               |
| -------------- | -------------------------------------------------- |
| 初始           | 用户名和密码为空                                   |
| 提交前校验失败 | 不发请求，显示必填提示                             |
| 提交中         | 按钮显示登录中，禁止重复点击                       |
| 登录成功       | 保存登录态并跳转 `/admin`                          |
| 业务失败       | 显示后端 `msg`                                     |
| 网络失败       | 显示「无法连接后端服务，请确认后端服务是否已启动」 |

### 后台首页

后台首页读取本地登录态中的当前管理员信息展示摘要。

如果用户刷新页面后本地只有 Token，没有当前管理员信息，页面应通过受保护接口重新拉取资料，或在进入管理员设置时完成资料加载。不能使用假管理员数据填充。

### 管理员资料

进入管理员设置页时：

```text
调用 GET /api/admin/profile
→ 成功：填充用户名、管理员名称、角色、头像地址
→ 失败：显示错误状态
```

保存资料时：

```text
校验 name 非空
→ 调 PUT /api/admin/profile
→ 成功：更新本地 currentUser，并显示「资料已保存」
→ 失败：显示后端 msg
```

字段规则：

| 字段       | 规则                                                                                                                        |
| ---------- | --------------------------------------------------------------------------------------------------------------------------- |
| 用户名     | 可编辑，不能为空                                                                                                            |
| 角色       | 只读                                                                                                                        |
| 管理员名称 | 可编辑，不能为空                                                                                                            |
| 头像图片   | 使用 `avatar` 字段作为图片地址预览和保存，可编辑，可为空；后端返回 `null`、空字符串或图片加载失败时，前端显示纯灰色默认头像 |

### 修改密码

流程：

```text
输入旧密码和新密码
→ 校验 oldPassword 非空
→ 校验 newPassword 长度 6-50 位
→ 调 PATCH /api/admin/profile/password
→ 成功：清理本地登录态，跳转 /admin/login
→ 失败：显示后端 msg
```

后端真实成功消息当前为「密码修改成功」。前端不能依赖固定成功文案判断流程，应以接口成功本身作为重新登录的触发条件。

### Token 状态

初始状态显示「未校验」。

校验 Token：

```text
点击「校验 Token」
→ 调 GET /api/admin/validate
→ 成功且 valid=true：显示「有效」
→ 401 或失败：显示后端 msg，并清理登录态
```

刷新 Token：

```text
点击「刷新 Token」
→ 调 POST /api/admin/refresh
→ 成功：保存新 token，显示「已刷新」
→ 401 或失败：显示后端 msg，并清理登录态
```

### 退出登录

退出登录入口放在后台顶部栏。

顶部栏当前管理员区域应展示当前管理员头像和名称。名称只显示管理员名称本身，不加「当前管理员：」等解释性前缀，也不在同一位置追加用户名。头像使用 `currentUser.avatar`；为空、`null` 或图片加载失败时显示纯灰色默认头像，不使用图标替代头像。

流程：

```text
点击「退出登录」
→ 打开确认弹窗
→ 确认后调 POST /api/admin/logout
→ 无论后端是否保存会话，前端清理 token 和 currentUser
→ 回到 /admin/login
```

退出登录应使用确认弹窗，避免误点。

## 路由守卫

规则：

| 场景                               | 结果                |
| ---------------------------------- | ------------------- |
| 未登录访问 `/admin`                | 跳转 `/admin/login` |
| 未登录访问 `/admin/settings/admin` | 跳转 `/admin/login` |
| 已登录访问 `/admin/login`          | 跳转 `/admin`       |
| 请求层收到 `code=401`              | 清理本地登录态      |

前端不能解析 JWT 来判断权限或过期时间。JWT 签发、解析、过期和密码版本判断属于后端职责。

## 后台基础组件

本阶段必须沉淀后台基础组件，禁止每个页面自建按钮、弹窗、提示和表单行。

### 通用基础组件

| 组件            | 职责                                                 |
| --------------- | ---------------------------------------------------- |
| `Button`        | 统一按钮视觉、加载、禁用、图标位置                   |
| `Input`         | 统一输入框、只读、错误状态                           |
| `FormField`     | 统一 label、输入控件和错误文案位置                   |
| `Alert`         | 展示成功、错误、警告、信息提示                       |
| `Modal`         | 通用弹窗容器                                         |
| `ConfirmDialog` | 危险或确认操作弹窗                                   |
| `LoadingState`  | 页面级或区块级加载                                   |
| `ErrorState`    | 加载失败和重试入口                                   |
| `EmptyState`    | 列表或区块无数据状态                                 |
| `PageHeader`    | 页面标题、说明、右侧操作区                           |
| `DataSection`   | 统一完整宽度操作面板，包含标题、描述、内容和底部操作 |
| `AdminAvatar`   | 统一后台头像图片、空值和加载失败占位                 |

组件职责边界：

- 公共组件通过 props 接收数据。
- 公共组件不直接调用后端接口。
- 公共组件不直接读写 Token。
- 公共组件不依赖具体业务模块。
- 公共组件不伪造后端数据。
- 页面组件负责连接接口和状态，公共组件负责展示。

### 后台布局组件

| 组件           | 职责                                     |
| -------------- | ---------------------------------------- |
| `AdminShell`   | 后台整体框架，包含侧边栏、顶部栏和内容区 |
| `AdminSidebar` | 后台菜单                                 |
| `AdminTopbar`  | 当前管理员信息和退出登录入口             |
| `AdminNavItem` | 单个后台菜单项                           |

页面组合规则：

| 页面                | 使用组件                                                                              |
| ------------------- | ------------------------------------------------------------------------------------- |
| `LoginView`         | `Button`、`Input`、`FormField`、`Alert`                                               |
| `DashboardView`     | `PageHeader`、`AdminAvatar`、`Button`                                                 |
| `AdminSettingsView` | `PageHeader`、`DataSection`、`FormField`、`Input`、`Button`、`Alert`、`ConfirmDialog` |

后续文章、分类、标签、评论等后台页面必须复用这些基础组件，不能再各自实现一套按钮、弹窗、错误提示和空状态。

## 组件视觉契约

### Button

支持属性：

```text
variant: primary / secondary / danger / ghost / link
size: sm / md
loading: boolean
disabled: boolean
icon?: ReactNode
iconPosition?: left | right
```

规则：

- 主按钮用于当前区块的主要动作。
- 危险按钮只用于退出、删除、危险确认。
- 主按钮使用深灰底色和清晰 hover 状态，不使用渐变或发光阴影。
- loading 时按钮不可重复点击。
- loading 时隐藏或替换 icon，但按钮尺寸不能跳动。
- 页面不直接写按钮内部布局。

### Input

支持属性：

```text
label
value
placeholder
error
disabled
readOnly
type: text / password
```

规则：

- 默认使用 1px 灰线边框。
- 聚焦时使用低饱和赤陶红边框和清晰焦点环。
- 错误时使用红色边框和错误文案。
- 只读字段使用浅灰底，不假装可编辑。
- 输入框高度、圆角、边框和聚焦阴影必须和按钮保持同一视觉体系。

### Modal 和 ConfirmDialog

规则：

- 弹窗居中显示。
- 使用背景遮罩。
- 遮罩必须覆盖整个文档高度，不能只覆盖首屏。
- 结构分为标题、正文、底部按钮三段。
- 危险确认按钮必须使用明确文案，例如「确认退出」「确认删除」。
- 弹窗组件不直接执行业务请求，只通过回调交给页面处理。

### Alert

支持类型：

```text
success / error / warning / info
```

规则：

- 使用紧凑文字、细边框或左侧语义线区分类型，不使用大面积彩色背景。
- 文案使用后端 `msg` 或前端明确错误。
- 不编造错误原因。
- 默认不显示图标，避免每条反馈都产生装饰噪声。

## 图标规则

图标只用于按钮、菜单、状态提示、空状态辅助符号等适合符号化表达的场景。不是所有按钮、字段或区块都必须带图标，文案已经清楚时可以不用图标。

需要展示头像、上传预览、文件图片预览等真实视觉对象时，必须使用图片、图片地址预览或明确的图片占位，不使用图标顶替。

图标必须来自统一 React 图标组件库。建议引入 `lucide-react`。

选择原因：

- 图标覆盖后台常见操作。
- 风格统一。
- 可以按组件导入。
- 适合按钮、菜单和状态提示。

禁止事项：

- 禁止在页面或组件里手写 SVG 图标。
- 禁止复制不明来源的 SVG path。
- 禁止为了一个按钮临时绘制 icon。
- 禁止使用 emoji 当图标。
- 禁止使用 emoji 当状态、菜单、按钮或提示的视觉符号。
- 禁止不同页面各自引入不同图标体系。
- 禁止把头像、上传预览、文件图片预览等图片场景改成图标。

允许事项：

- 从统一图标库导入图标组件。
- 在 `Button`、`AdminNavItem`、`Alert`、`EmptyState` 等基础组件中通过 props 传入 icon。
- 图标尺寸统一由组件控制。
- 找不到合适图标时，宁可不用图标，也不手写 SVG，不用 emoji 顶替。
- 图片场景可以使用后端返回 URL、本地图片资源、纯色默认头像或明确图片占位。

建议映射：

| 场景       | 图标                        |
| ---------- | --------------------------- |
| 后台首页   | `Home` 或 `LayoutDashboard` |
| 管理员设置 | `UserCog` 或 `Settings`     |
| 登录       | `LogIn`                     |
| 保存资料   | `Save`                      |
| 修改密码   | `KeyRound`                  |
| 校验 Token | `ShieldCheck`               |
| 刷新 Token | `RefreshCw`                 |
| 退出登录   | `LogOut`                    |
| 成功提示   | `CheckCircle`               |
| 错误提示   | `CircleAlert`               |
| 警告提示   | `TriangleAlert`             |
| 信息提示   | `Info`                      |

如果后续实现引入 `lucide-react`，必须同步更新 `package.json` 和 `package-lock.json`。

## 第一版实现拆分确认

第一版后台实现按以下工程拆分落地：

- 公共 UI 组件放在 `src/components/ui/`，页面不得重复实现按钮、输入框、弹窗、提示、加载、错误和空状态。
- 后台框架组件放在 `src/components/admin/`，`AdminLayout` 只负责组合后台框架和路由出口。
- 图标继续统一使用 `lucide-react`，但只保留真正帮助识别的菜单和操作图标；头像、上传预览、文件图片预览等场景必须使用图片或图片占位。
- 新增 `AdminAvatar` 统一顶部栏、后台首页和管理员设置页的头像与失败占位逻辑。
- 使用 `@fontsource-variable/noto-sans-sc` 在项目内加载 `Noto Sans SC Variable`，不依赖外部字体服务。
- `AdminShell` 使用固定侧边栏、固定顶部栏和独立内容滚动区，删除顶部栏自动隐藏逻辑。
- 后台首页使用横向管理员概览面板，管理员设置使用单栏完整宽度操作面板，不给章节标题配装饰图标。
- 文章模块完成前隐藏侧边栏入口，不修改文章 API、Query、编辑器、列表页或编辑页实现。
- 登录接口以真实后端 `AdminVO` 平铺结构为准，前端从 `data.token` 保存 Token，从同一个 `data` 中拆出当前管理员资料。
- 第一版允许按页面质量需要引入外部 UI 组件库、局部组件依赖和动画库，不要求全部手写。
- 可以引入 GSAP 或其他适合 React 的动效依赖；动画必须服务真实交互反馈或页面质感，不能遮挡数据、影响表单操作或造成布局跳动。

## 数据与错误状态

所有页面必须覆盖以下状态：

| 状态         | 要求                                               |
| ------------ | -------------------------------------------------- |
| 初始         | 表单和页面默认状态明确                             |
| 加载中       | 禁止重复提交，区块显示加载                         |
| 成功         | 更新界面和本地状态                                 |
| 业务失败     | 展示后端 `msg`                                     |
| 网络失败     | 展示「无法连接后端服务，请确认后端服务是否已启动」 |
| 未登录或过期 | 清理登录态并回到登录页                             |

管理员设置页没有“空数据”主场景。后续列表页必须使用 `EmptyState`。

## 测试验收

测试不验证颜色、间距、阴影和具体 SVG 内容，只验证业务行为、状态和可访问文案。

### 接口与类型

- `login` 按后端平铺 `AdminVO` 返回解析 Token。
- `getCurrentAdminProfile` 调 `GET /admin/profile`。
- `updateAdminProfile` 调 `PUT /admin/profile`。
- `changeAdminPassword` 调 `PATCH /admin/profile/password`。
- `validateAdminToken` 调 `GET /admin/validate`。
- `refreshAdminToken` 调 `POST /admin/refresh`。
- `logout` 调 `POST /admin/logout`。

### 登录页

- 用户名和密码为空时，不发登录请求并展示必填提示。
- 登录成功后，保存 token 和 currentUser，并跳转 `/admin`。
- 登录失败时，展示后端 `msg`。
- 登录中不能重复提交。

### 管理员设置页

- 进入页面会请求当前管理员资料。
- 资料加载成功后，表单显示真实后端字段。
- 保存资料成功后，更新本地 `currentUser`。
- 保存资料失败时，展示后端 `msg`。
- 新密码长度不足 6 位时，不发请求并展示提示。
- 修改密码成功后，清理登录态并跳转登录页。
- 校验 Token 成功时显示有效。
- 刷新 Token 成功时保存新 Token。
- 401 时清理登录态。

### 路由

- 未登录访问管理员设置页会跳转 `/admin/login`。
- 已登录访问登录页会跳转 `/admin`。

### 组件

- `Button` 在 loading 时禁用并保持文案或加载状态可见。
- `ConfirmDialog` 点击取消不执行确认回调。
- `ConfirmDialog` 点击确认执行确认回调。
- `Alert` 能显示传入的错误或成功文案。
- `AdminAvatar` 在头像为空或图片加载失败时显示纯灰色占位。
- 后台侧边栏不显示尚未完成的文章管理入口。
- 图标不作为测试断言主体，不测试 SVG 细节。

## 非目标

本阶段不做：

- 管理员注册。
- 找回密码。
- 多管理员列表。
- 权限角色管理。
- 文章统计。
- 文章管理。
- 分类和标签管理。
- 评论、留言、友链管理。
- 文件上传。
- 前台页面视觉设计。

## 实施边界

后续进入实现时应遵守：

- 页面放在 `src/views/admin/`。
- 后台布局放在 `src/layout/` 或后台布局组件目录。
- 公共组件放在 `src/components/`。
- 接口方法放在 `src/api/`。
- 类型放在 `src/types/`。
- 请求仍通过 `src/utils/request.ts`。
- 登录态仍由 `src/store/auth.ts` 管理。
- 不在页面中直接写 Axios。
- 不在页面中直接拼接完整后端地址。
- 不在页面中解析 JWT。
