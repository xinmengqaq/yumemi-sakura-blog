# 09-开发顺序与验收标准

开发顺序按“先基础能力，再内容模块，再互动模块，再统计优化”推进。

本文中的“测试”默认指后端开发式测试，也就是后端开发者为了确认当前实现能跑通、边界不明显破坏而做的自查动作，例如运行已有测试类、补充小范围单元测试、执行 `./mvnw clean test`、用接口工具或日志检查一次关键链路。它不等同于测试工程师负责的系统测试、专项测试、全量回归测试或完整测试计划。开发节奏不按“大阶段结束后统一补测”执行，而是按“小闭环完成后立刻自查”执行。一个小闭环指一段可以独立验证的后端能力，例如 Mapper SQL 跑通、工具类完成一个核心方法、Service 完成一个业务分支、Controller 完成一个接口入口。

数据库学习内容必须覆盖常用 SQL 能力，但只作为后端开发过程中的数据库练习和 Mapper 编写依据，不单独进入测试工程师职责。文章相关阶段要覆盖 `INSERT`、`DELETE`、`UPDATE`、`SELECT` 四类核心操作；覆盖 `WHERE`、`ORDER BY`、`GROUP BY`、`HAVING`、`LIMIT`、`DISTINCT`、`JOIN` 等常用查询语法；统计阶段覆盖 `COUNT`、`SUM`、`AVG`、`MAX`、`MIN` 等聚合函数。每个 SQL 点都要结合真实博客业务场景学习，不写脱离项目的孤立练习表。

## 第一阶段：项目基础改造

目标：

- 从 JPA 持久层切到 MyBatis。
- 建好单模块过渡目录结构，为后续 Maven 多模块重构做准备。
- 建好统一返回、异常、校验、分页、日志、JWT 配置等基础能力。
- 确定 Service 接口与 ServiceImpl 分层，不再使用单一 Service 类直连持久层。

开发内容：

- 引入 MyBatis。
- 引入 PageHelper。
- 使用 Logback + SLF4J。
- 先在单模块内规范化 `service / service/impl / mapper / dto / vo / utils / config / interceptor` 目录。
- 移除或停用 `repository` 层。
- 配置 MyBatis XML 路径。
- 引入 JJWT 依赖。
- 配置 JWT 参数。
- 建好 JWT 配置读取类和基础工具类。
- 配置全局异常处理。
- 定义统一分页返回结构。
- 配置 SQL 调试日志开关。

验收标准：

- 项目能正常启动。
- Swagger 能正常访问。
- 数据库连接正常。
- MyBatis 查询能跑通。
- 异常能统一返回 `Result`。
- JWT 配置能被 Java 配置类正常读取。
- JWT 工具类能完成 Token 签发和解析。
- 业务代码可以使用 `@Slf4j` 输出控制台日志。
- 需要排查 SQL 时，可以通过配置开关打开 SQL 日志。

小闭环开发自查要求：

- MyBatis 基础查询跑通后，立刻补充 Mapper 查询单元测试或轻量集成测试。
- JWT 工具类每完成签发、解析、过期、非法 Token 一个能力，就立刻补充对应单元测试。
- 统一异常返回链路跑通后，立刻补充异常返回测试。
- 每个小闭环开发自查通过后再继续下一个基础能力；阶段结束前执行一次 `./mvnw clean test`。

## 第二阶段：管理员登录验证与资料维护

目标：

- 完成单人后台登录。
- JWT 能保护后台接口。
- 管理员资料能维护。

开发内容：

- 管理员登录。
- JWT 签发。
- JWT 校验。
- JWT 拦截器。
- JWT 拦截器注册配置。
- 校验 Token 接口。
- 刷新 Token 接口。
- 确认管理员数据手动入库方案。
- 确认 `admin.password_version` 字段。
- 获取当前管理员信息。
- 修改管理员资料。
- 修改密码。
- 密码加密存储。

验收标准：

- 不提供管理员注册接口。
- 已手动入库的管理员可以正常登录。
- 密码不会明文入库。
- 登录成功返回 Token。
- Token 错误或过期不能访问后台。
- 未登录访问后台接口会被拦截。
- 校验 Token 接口能在 Token 有效时返回 `valid=true`。
- 刷新 Token 接口能在旧 Token 有效时返回新 Token。
- 修改密码后旧 Token 应失效。
- 当前管理员信息接口不会返回密码。

小闭环开发自查要求：

- 管理员登录 Service 跑通后，立刻补充登录成功、密码错误、账号不存在测试。
- JWT 拦截器完成未登录拦截后，立刻补充未登录测试；完成 Token 错误和过期处理后，再分别补充对应测试。
- 校验 Token 接口完成后，立刻补充有效 Token 和无效 Token 测试。
- 刷新 Token 接口完成后，立刻补充刷新成功和旧 Token 无效测试。
- 修改密码闭环完成后，立刻补充旧 Token 失效测试。
- 当前管理员信息闭环完成后，立刻补充不返回密码测试。
- 每个小闭环开发自查通过后再继续下一个管理员能力；阶段结束前执行一次 `./mvnw clean test`。

## 第三阶段：Maven 多模块重构与工程拆分

目标：

- 在进入文章模块设计前，先把工程结构改造成更适合学习和扩展的多模块结构。
- 学会 Maven 多模块项目的父子工程、依赖边界和模块职责划分。
- 按“分层职责”和“业务模块”建立清晰边界，避免后续文章、说说、相册等功能堆在一个包里。

开发内容：

- 引入 Maven 父工程与子模块结构。
- 拆分公共基础模块。
- 拆分启动模块。
- 拆分业务模块。
- 拆分持久层相关结构。
- 梳理 `controller / service / service/impl / mapper / entity / dto / vo / config / interceptor / common / utils` 在多模块下的放置规则。
- 调整 Maven 依赖关系，避免模块之间循环依赖。
- 调整 Spring Boot 启动类扫描范围。
- 调整 MyBatis Mapper 扫描范围。
- 调整资源文件、Mapper XML、配置文件加载路径。
- 保证已有管理员登录验证与管理员资料能力在多模块结构下仍能运行。

验收标准：

- 根目录可以通过 Maven Wrapper 完成完整构建。
- 启动模块可以正常启动 Spring Boot。
- 公共返回、异常、JWT、日志、分页等基础能力可以被业务模块复用。
- 管理员登录、JWT 拦截功能在多模块结构下不退化。
- 模块之间依赖方向清晰，不出现业务模块互相乱引用。
- 新增文章模块时，有明确的代码放置位置。

小闭环开发自查要求：

- 每迁移完一个已有测试目录或测试类，立刻运行对应测试，确认多模块拆分没有破坏原能力。
- 启动模块能加载公共能力与业务 Bean 后，立刻补充模块依赖和启动上下文测试。
- Mapper 扫描、XML 加载、配置读取分别跑通后，立刻补充对应集成验证。
- 每个小闭环开发自查通过后再继续下一项拆分；阶段结束前在根目录执行一次 `./mvnw clean test`。

## 第四阶段：文章模块设计与基础增删改查

目标：

- 以文章为第一个完整业务模块，学习从接口到数据库的基础 CRUD 闭环。
- 先只处理文章主表，不急着引入分类、标签等多表关系。
- 掌握 Controller、DTO、Service、Mapper、XML、VO 的完整协作链路。

开发内容：

- 设计文章模块目录结构。
- 设计文章主表字段。
- 编写 `article` 表建表 SQL，落地字段类型、非空约束、默认值和状态取值约束。
- 明确文章模块按单管理员后台设计，`article` 表不保存 `admin_id`，本阶段暂不启用分类、标签外键依赖。
- 为文章基础查询准备必要索引，例如状态、创建时间、发布时间。
- 准备文章主表的本地开发库示例数据，覆盖草稿、已发布、隐藏等基础状态，示例 SQL 文件统一放在根目录 `sql/` 下。
- 编写文章 Entity。
- 编写文章新增 DTO。
- 编写文章修改 DTO。
- 编写文章详情 VO。
- 新增文章。
- 修改文章。
- 删除文章。
- 查询文章详情。
- 查询文章基础列表。
- 在文章主表 SQL 中覆盖 `INSERT INTO ... VALUES ...`、`UPDATE ... SET ... WHERE ...`、`DELETE FROM ... WHERE ...`、`SELECT ... FROM ... WHERE ...` 四类核心操作。
- 使用 MyBatis XML 完成文章主表 SQL。
- 给文章基础接口补齐 Swagger 注解。

验收标准：

- 可以新增一篇文章。
- 可以修改文章标题、摘要、正文、封面、状态等基础字段。
- 可以删除文章。
- 可以按 ID 查询文章详情。
- 可以查询文章基础列表。
- Controller 不写业务逻辑。
- ServiceImpl 完成业务编排。
- Mapper 只负责数据库访问。
- MyBatis XML SQL 可读、可维护。

小闭环开发自查要求：

- 文章主表 Mapper SQL 跑通后，立刻补充 Mapper 层文章主表 SQL 测试。
- 新增、修改、删除、详情、基础列表每完成一个接口闭环，就立刻补充对应 Service 或 Controller 测试。
- 必填字段、ID 不存在、非法状态等错误分支完成后，立刻补充对应错误场景测试。
- 每个小闭环开发自查通过后再继续下一个文章基础能力；阶段结束前执行一次 `./mvnw clean test`。

## 第五阶段：文章分类标签与数据库多表关系

目标：

- 在文章基础 CRUD 之后，再学习数据库多表关系。
- 掌握一对多、多对多关系在博客文章中的落地方式。
- 明确分类、标签、文章标签关联表之间的业务边界。

开发内容：

- 分类管理。
- 标签管理。
- 编写 `category`、`tag`、`article_tag` 建表 SQL，落地唯一约束、非空约束和更新时间字段。
- 明确 `article.category_id`、`article_tag.article_id`、`article_tag.tag_id` 的外键关系和删除策略。
- 为分类排序、标签去重、文章按分类筛选、文章按标签筛选补充必要索引。
- 准备分类、标签、文章标签关联的本地开发库示例数据，覆盖一对多和多对多关系。
- 文章绑定分类。
- 文章标签关联。
- 文章修改时同步维护标签关联。
- 查询文章详情时返回分类信息。
- 查询文章详情时返回标签列表。
- 查询文章列表时支持分类筛选。
- 查询文章列表时支持标签筛选。
- 设计文章表、分类表、标签表、文章标签关联表之间的 SQL。
- 在文章详情和列表 SQL 中学习 `JOIN ... ON ...`，把文章、分类、标签、文章标签关联表连接起来。
- 在标签列表或文章标签查询中学习 `SELECT DISTINCT ...`，避免多表关联后返回重复标签或重复文章。
- 处理分类删除前的文章占用校验。
- 处理标签重复校验。
- 处理文章重复绑定同一标签的校验。

验收标准：

- 分类不存在时不能保存文章。
- 分类下有文章时不能删除分类。
- 标签不能重复。
- 同一篇文章不能重复关联同一标签。
- 文章详情能正确返回分类和标签。
- 文章列表能按分类筛选。
- 文章列表能按标签筛选。
- 修改文章标签后，关联表数据和最终查询结果一致。

小闭环开发自查要求：

- 分类新增、修改、删除占用校验每完成一个小闭环，就立刻补充对应测试。
- 标签重复校验、文章重复绑定标签校验完成后，立刻补充对应测试。
- 文章详情返回分类、标签跑通后，立刻补充详情组合返回测试。
- 分类筛选、标签筛选与分页组合跑通后，立刻补充组合查询测试。
- Mapper 集成测试统一使用 H2 内存数据库，在测试夹具中创建 `article`、`category`、`tag`、`article_tag` 完整临时关系表和隔离数据；不得连接或改动本地 PostgreSQL 开发库。
- 每个小闭环开发自查通过后再继续下一个多表关系能力；阶段结束前执行一次 `./mvnw clean test`。

## 第六阶段：文章状态流转与事务管理

目标：

- 在文章已经涉及多表写入后，再学习 Spring 事务管理。
- 掌握文章发布、草稿、隐藏、删除、置顶、推荐等状态变化。
- 保证文章主表和文章标签关联表的写入要么一起成功，要么一起失败。

开发内容：

- 文章发布。
- 文章草稿。
- 文章隐藏。
- 文章置顶。
- 文章推荐。
- 发布文章时自动记录发布时间。
- 确认发布、隐藏、置顶、推荐等状态变化在数据库字段上的存储规则。
- 修改文章时同步更新文章标签关联。
- 删除文章时在同一个 Service 事务中同步删除文章标签关联和文章主记录。
- 批量删除文章，学习列表页多选操作、`ids` 数组参数、批量删除 SQL 和多表清理事务。
- 使用 `@Transactional` 管理文章新增、修改、删除等多表写操作。
- 验证事务回滚场景。
- 梳理事务方法的放置位置和调用方式。

验收标准：

- 草稿文章前台不可见。
- 隐藏文章前台不可见。
- 发布文章自动记录发布时间。
- 新增文章时，文章主表和标签关联表写入一致。
- 修改文章时，文章主表和标签关联表更新一致。
- 删除文章时，文章标签关联同步清理。
- 批量删除文章时，当前阶段同步清理文章标签关联并返回实际删除数量；评论模块完成后，再同步清理真实评论记录，`comment_count` 字段本身不代表评论记录可删除。
- 多表写入中途失败时，数据库不会留下半成功数据。
- 事务只放在 Service 层业务方法上，不放在 Controller 或 Mapper 上。

小闭环开发自查要求：

- 草稿、发布、隐藏、置顶、推荐每完成一个状态流转闭环，就立刻补充对应测试。
- 发布时自动记录发布时间跑通后，立刻补充发布时间测试。
- 文章主表和标签关联表多表写入闭环完成后，立刻补充写入一致性测试。
- 批量删除文章跑通后，立刻补充批量删除成功、空 `ids`、关联数据同步清理测试。
- 事务回滚场景设计并跑通后，立刻补充失败时不留下半成功数据的测试。
- 每个小闭环开发自查通过后再继续下一个状态或事务能力；阶段结束前执行一次 `./mvnw clean test`。

## 第八阶段：文章模块 Spring Boot AOP 实装

目标：

- 在文章模块功能基本稳定后，再引入 AOP 学习横切能力。
- 通过真实业务场景学习日志、操作记录、耗时统计等横切逻辑。
- 避免把日志和统计逻辑散落到每个业务方法里。
- 结合 Spring Boot 4 学习 AOP 自动配置、代理选择和切面执行顺序等机制，而不是只会编写切面注解。

开发内容：

- 定义文章行为日志注解。
- 确认 AOP 依赖进入类路径后，Spring Boot 4 如何通过自动配置启用 Spring AOP，并理解通常不需要手写 `@EnableAspectJAutoProxy`。
- 学习 Spring Boot 4 默认使用 CGLIB 代理的行为，以及通过 `spring.aop.proxy-target-class=false` 切换为 JDK 动态代理的条件和差异。
- 对比 JDK 动态代理与 CGLIB 代理，理解接口代理、类代理、目标对象和代理对象之间的关系。
- 使用 Spring AOP 拦截文章新增、修改、删除、发布、隐藏等操作。
- 记录操作类型、操作对象 ID、操作结果、耗时。
- 对文章详情或列表查询记录接口耗时日志。
- 学习切点表达式、通知类型、连接点参数和 `ProceedingJoinPoint`，明确它们在文章业务日志中的职责。
- 验证同类内部方法调用不会经过代理的场景，理解 `this` 自调用导致切面失效的原因，并避免依赖自注入等不必要的绕行方案。
- 学习 `@Order` 或 `Ordered` 对多个切面的排序作用，并观察文章切面与事务拦截器的进入、返回和异常执行顺序。
- 处理 AOP 中的异常透传。
- 明确 AOP 与全局异常处理、业务事务之间的执行关系。

验收标准：

- 文章新增、修改、删除等关键操作能输出统一行为日志。
- AOP 不改变原有业务接口返回结果。
- 业务异常仍由全局异常处理统一返回。
- 事务回滚时不会误判为成功操作。
- 日志内容能定位操作类型和目标文章。
- 查询耗时日志不会影响分页接口正常返回。
- 不手写 `@EnableAspectJAutoProxy` 时，Spring Boot 4 的 AOP 自动配置仍能正常生效。
- 能通过配置切换并验证 CGLIB 代理与 JDK 动态代理，说明当前文章 Service 最终采用哪种代理及原因。
- 能说明并通过验证用例复现同类内部调用不触发切面的代理边界。
- 多个切面或切面与事务同时存在时，执行顺序明确，异常透传和事务回滚结果符合预期。

小闭环开发自查要求：

- 文章新增、修改、删除、发布、隐藏每接入一个 AOP 行为闭环，就立刻补充对应测试或验证用例。
- AOP 接入第一个业务接口后，立刻补充不改变接口返回结果测试。
- Spring Boot 4 自动配置生效后，立刻验证无需手写 `@EnableAspectJAutoProxy` 也能创建代理并执行切面。
- CGLIB 与 JDK 动态代理切换跑通后，立刻补充代理类型和业务行为验证。
- 同类内部调用场景复现后，立刻补充“外部代理调用生效、内部 `this` 调用不生效”的对比验证。
- 业务异常透传和全局异常处理协作跑通后，立刻补充协作测试。
- 事务回滚时不记录成功操作的规则跑通后，立刻补充回滚验证。
- 切面顺序与事务顺序确认后，立刻补充正常返回和异常回滚两条顺序验证。
- 每个小闭环开发自查通过后再继续下一个 AOP 能力；阶段结束前执行一次 `./mvnw clean test`。

## 第十三阶段：统计与后台首页

目标：

- 做一个够用的后台概览。

开发内容：

- 内容数量统计。
- 今日访问统计。
- 总访问统计。
- 热门文章。
- 最新评论。
- 最新留言。
- 最新友链申请。
- 在统计 SQL 中学习 `COUNT(*)`，统计文章、说说、评论、留言等内容数量。
- 在文章统计 SQL 中学习 `SUM(view_count)`，统计文章总阅读量。
- 在文章统计 SQL 中学习 `AVG(view_count)`，统计文章平均阅读量。
- 在文章统计 SQL 中学习 `MAX(view_count)` 和 `MIN(view_count)`，查询最高阅读量和最低阅读量。
- 在内容统计 SQL 中学习 `GROUP BY`，按文章状态、分类或评论状态分组统计。
- 在分组统计 SQL 中学习 `HAVING`，筛选文章数量或平均阅读量达到条件的分组。

验收标准：

- 后台首页能看到核心数据。
- 统计接口不影响正常内容接口。

- 最新评论、留言按时间倒序。

小闭环开发自查要求：

- 内容数量、今日访问、总访问统计每完成一个统计闭环，就立刻补充对应测试。
