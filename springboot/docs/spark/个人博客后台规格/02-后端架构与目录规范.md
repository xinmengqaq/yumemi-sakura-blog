# 02-后端架构与目录规范

## 总体架构

后端采用 Spring Boot 多模块单体项目。

多模块只用于学习 Maven 工程拆分、约束代码边界和降低包结构混乱，不拆成微服务，不引入远程调用，不改变“一次启动一个后端服务”的部署方式。

业务代码仍保持三层架构：

```text
controller -> service interface -> service impl -> mapper -> database
```

项目从当前 JPA Repository 方案迁移为 MyBatis Mapper 方案。

多模块后的依赖方向：

```text
springboot-web -> springboot-admin
springboot-web -> springboot-article
springboot-web -> springboot-common
springboot-admin -> springboot-common
springboot-article -> springboot-common
```

规则：

- `springboot-web` 是唯一启动模块，放启动类、全局配置和 `application.yml`。
- `springboot-common` 放公共返回、异常、分页、枚举、通用工具和基础配置属性。
- `springboot-admin` 放认证、管理员资料、JWT 校验所需的业务代码。
- `springboot-article` 放文章、分类、标签、文章标签关联。
- 后续说说、相册、评论、留言、友链等可以继续按业务域拆成独立 Maven 模块。
- 业务模块之间默认不互相依赖，需要复用的内容上沉到 `springboot-common`。
- 多模块工程仍由根目录 Maven Wrapper 统一构建和测试。

## 技术选择

| 能力 | 选择 |
|---|---|
| Web 框架 | Spring Boot WebMVC |
| 数据库 | PostgreSQL |
| 持久层 | MyBatis |
| 分页 | PageHelper |
| 接口文档 | Springdoc OpenAPI / Swagger |
| 日志 | Spring Boot 默认 Logback |
| 横切能力学习 | Spring AOP |
| 工具注解 | Lombok |
| 认证 | JJWT |

Maven 版本统一由父工程管理，子模块只声明自己实际需要的依赖。

分页不自研，使用 `com.github.pagehelper:pagehelper-spring-boot-starter`。

JWT 相关依赖统一使用 JJWT 官方拆分坐标：`jjwt-api`、`jjwt-impl`、`jjwt-jackson`。

## Maven 多模块结构

```text
springboot
├── pom.xml
├── springboot-common
│   ├── pom.xml
│   └── src/main/java/com/xinmengqaq/springboot/common
├── springboot-admin
│   ├── pom.xml
│   ├── src/main/java/com/xinmengqaq/springboot/admin
│   └── src/main/resources/mapper/admin
├── springboot-article
│   ├── pom.xml
│   ├── src/main/java/com/xinmengqaq/springboot/article
│   └── src/main/resources/mapper/article
└── springboot-web
    ├── pom.xml
    ├── src/main/java/com/xinmengqaq/springboot/SpringbootApplication.java
    └── src/main/resources/application.yml
```

根 `pom.xml` 职责：

- `packaging` 使用 `pom`。
- 通过 `modules` 声明所有子模块。
- 通过 `dependencyManagement` 统一管理 Spring Boot、MyBatis、PageHelper、JJWT 等版本。
- 统一 Java 版本为 21。

启动模块 `springboot-web` 职责：

- 持有唯一 Spring Boot 启动类。
- 持有 `application.yml`。
- 持有 Swagger、WebMvc、拦截器注册、跨域、MyBatis 扫描等全局配置。
- 依赖各业务模块。

公共模块 `springboot-common` 职责：

- 不依赖具体业务模块。
- 不放 Controller、Mapper、业务 Service。
- 提供所有业务模块都能复用的基础能力。

业务模块职责：

- 每个业务模块内部继续按三层结构拆分。
- Controller、Service、Mapper、Entity、DTO、VO 优先放在业务模块内部。
- Mapper XML 跟随业务模块放在该模块自己的 `resources/mapper/模块名` 下。

## controller

Controller 只负责：

- 接收 HTTP 请求。
- 接收 DTO 参数。
- 调用 Service 接口。
- 返回 Result。
- 编写 Swagger 注解。

Controller 不做：

- 不写业务判断。
- 不操作 Mapper。
- 不拼 SQL。
- 不直接返回 Entity。
- 不写密码加密、JWT 解析逻辑。

## service

Service 层采用接口和实现分离：

```text
springboot-admin/.../service/AdminService.java
springboot-admin/.../service/impl/AdminServiceImpl.java
springboot-article/.../service/ArticleService.java
springboot-article/.../service/impl/ArticleServiceImpl.java
```

规则：

- Controller 面向 Service 接口编程，不直接依赖实现类。
- `service` 包放业务接口定义。
- `service.impl` 包放业务实现类。
- `@Service` 标在实现类上，不标在接口上。
- `@Transactional` 优先放在实现类的公开业务方法上。
- Service 接口定义能力边界，ServiceImpl 负责具体业务编排。

ServiceImpl 负责：

- 编排业务流程。
- 做业务校验。
- 调用 Mapper。
- 调用 Utils。
- 控制事务。
- 处理状态流转。
- 维护关联关系。

需要事务的场景：

- 新增文章并保存标签关系。
- 删除文章并删除标签关联、评论。
- 删除说说并删除图片、评论。
- 删除相册并删除照片。
- 审核评论并更新评论数量。

当前已实现代码的事务边界：

- 管理员资料修改、管理员密码修改。
- 分类修改、分类删除。
- 标签修改、标签删除。
- 文章修改。
- 上述事务统一放在 ServiceImpl 的公开业务方法上，使用 `@Transactional(rollbackFor = Exception.class)`，保证运行时异常和受检异常都会触发回滚；异常不能在事务方法内被捕获后吞掉。
- 管理员密码修改需要锁定当前管理员记录，保证旧密码校验与密码更新不会被并发修改穿透。
- 分类、标签删除需要先锁定待删除记录，再执行占用校验和删除；数据库外键限制仍作为最终一致性防线。
- 当前分类新增、标签新增、文章新增、文章删除都只有一次写操作，不提前添加事务；唯一性和外键完整性分别由数据库唯一约束和外键约束保证。
- 当前纯查询方法不添加只读事务，避免在没有多次一致性读取需求时扩大事务范围。

## mapper

Mapper 负责数据库访问。

规则：

- Java Mapper 接口放在 `mapper` 包。
- XML 放在业务模块自己的 `resources/mapper/模块名`。
- 一个 Mapper 对应一个 XML。
- SQL 优先写 XML，便于学习增删改查和多表查询。
- Mapper 不写业务判断。
- Mapper 只给 ServiceImpl 调用，不给 Controller 直接调用。
- MyBatis XML 不要求所有字段都一股脑写死在 SQL 中；新增、修改、查询条件、批量关联等场景，可以按实际业务使用 `<if>`、`<foreach>`、`<set>`、`<trim>` 等动态 SQL 标签。
- MyBatis XML 中重复出现的真实字段列优先抽成 `<sql>` 片段，并通过 `<include>` 复用；例如分类列表中 `select` 和 `group by` 使用同一组 `categoryColumns`，避免查询字段和分组字段不一致。
- 可复用字段片段只放数据库真实字段列，例如 `c.id`、`c.name`、`c.updated_at`；不要把 `count(...)`、`case when ...`、带业务别名的计算字段放进这类字段片段，避免 `group by <include>` 时把聚合或计算逻辑错误带入分组。
- 修改语句优先只处理本次允许修改或确实传入的字段，避免无意义覆盖旧值；状态流转、发布时间、更新时间这类有业务含义的字段，仍由 Service 先判断业务规则，再交给 Mapper 写入。

国内后端编码思想统一总结：

- 复用不重复。同一逻辑出现 2 次及以上才抽取复用；XML 中条件、列、赋值重复出现时抽成 SQL 片段，Java 中业务逻辑重复出现时抽成公共方法、组件或工具类。一次性逻辑不要为了“看起来高级”提前抽象，避免制造不必要的跳转和维护成本。
- 动态按需。XML 条件传了才拼，不传就忽略；Java 参数可选时不强迫前端补无意义字段，业务分支按真实类型和状态执行。核心是传什么用什么，不传不影响，不制造无意义逻辑。
- 安全防注入。XML 中用户输入一律走预编译占位符，模糊查询也不能直接拼接用户字符串；Java 中请求参数要做格式校验、业务校验和敏感数据脱敏。任何来自外部的输入都默认不可信。
- 明确不偷懒。XML 禁止 `select *`，查什么列写什么列；Java 禁止直接把 Entity 返回给前端，使用 VO 精确控制响应字段。要什么拿什么，拿什么给什么，不多拿也不多给。
- 结果要确定。XML 列表查询必须有符合业务语义的 `ORDER BY`，必要时用主键兜底；Java 方法返回值要稳定，列表查询不返回 `null`，写操作要考虑幂等，并发场景要有锁、唯一约束或乐观锁思路。同样输入不应该产生随机结果。
- 优雅不硬编码。XML 使用 MyBatis 动态 SQL 能力处理条件前缀、末尾逗号和集合拼接，避免 `where 1=1`；Java 使用框架提供的校验、异常处理、事务、配置绑定等能力，避免到处手写重复样板代码。让框架处理通用脏活，人只写业务规则。
- 命名即文档。XML 片段 id、结果映射和参数名要表达业务含义；Java 类名、方法名、变量名要见名知义。不要使用 `sql1`、`getData`、`num` 这类让人猜含义的名字。
- 层层做防御。XML 字符串判断要同时处理 `null` 和空串，不假设上层一定传合法值；Controller 校验格式，Service 校验业务，Mapper 保证 SQL 拼接安全。每一层都为自己的输入负责。
- 事务保一致。XML 多表写入涉及的条件和值要保持一致；Java 多步写操作要使用事务，要么全部成功，要么全部回滚，事务方法内不能吞掉会影响回滚的异常。数据一致性优先于局部成功。
- 统一规范。XML 片段命名、缩进和 SQL 风格要统一；Java 接口路径、响应结构、异常处理、日志格式和分层职责要统一。目标是团队里不同人写出的代码像一个人写的，方便接手和排查。
- 一句话原则：2 次及以上才抽取复用、动态按需、安全防注入、明确不偷懒、结果要确定、优雅不硬编码、命名即文档、层层做防御、事务保一致、统一规范。

## entity

Entity 对应数据库表字段。

规则：

- Entity 用于 Mapper 查询结果承载。
- Entity 不直接作为请求参数。
- Entity 不直接作为接口返回对象。
- Entity 不写业务方法。

例如 `Admin` 实体允许有 `password` 字段，但返回前端必须使用不含密码的 VO。

## dto

DTO 用于接收前端请求参数。

命名示例：

```text
ArticleCreateDTO
ArticleUpdateDTO
ArticleQueryDTO
AdminLoginDTO
AdminRegisterDTO
CommentReviewDTO
SiteConfigUpdateDTO
```

规则：

- 新增和修改分开。
- 查询条件可以使用 QueryDTO。
- DTO 做基础参数校验。
- DTO 不代表数据库结构。

## vo

VO 用于返回前端。

命名示例：

```text
ArticleListVO
ArticleDetailVO
AdminVO
AlbumDetailVO
DashboardStatVO
```

规则：

- 列表 VO 少字段。
- 详情 VO 多字段。
- 不返回密码。
- 不暴露内部字段。
- 可以组合分类名、标签名、评论数等前端展示字段。

### 第五阶段对象落地约定

`springboot-article` 在第五阶段先落对象层，类名固定如下：

```text
entity:
Article
Category
Tag
ArticleTag

dto:
ArticleDTO
ArticlePageQueryDTO
CategoryCreateDTO
CategoryUpdateDTO
CategoryQueryDTO
TagCreateDTO
TagUpdateDTO
TagQueryDTO

vo:
ArticleVO
ArticleListVO
ArticleDetailVO
CategoryVO
TagVO
```

约定说明：

- 文章新增和修改参数当前继续沿用 `ArticleDTO`，其中 `categoryId` 和 `tagIds` 作为第五阶段多表关系输入字段。
- 文章列表筛选继续沿用 `ArticlePageQueryDTO`，在第五阶段补 `categoryId`、`tagId` 条件字段。
- 分类和标签新增、修改分别使用独立 DTO，不和数据库 Entity 混用。
- 文章返回对象在第五阶段补齐 `categoryName`、`tags` 等联表展示字段，同时保留第四阶段已有 `ArticleVO`，避免当前代码提前联动改动。

第七阶段文章分页、筛选与排序继续沿用 `ArticleController -> ArticleService -> ArticleServiceImpl -> ArticleMapper` 三层，不为单条文章列表查询额外建立独立 Query 三层。查询参数复用 `ArticlePageQueryDTO`，列表元素复用 `ArticleListVO`，分页外层使用公共 `PageResult`。

前台首页与文章阅读使用独立且集中的公开链路：`PublicContentController -> PublicContentService -> PublicContentServiceImpl -> PublicContentMapper`。该链路统一承接首页、公开文章、公开分类、公开标签、文章归档和匿名点赞，不按资源重复拆分多套三层；公开 SQL 必须固定过滤 `article.status = 'published'`，分类接口还必须过滤 `category.status = 'visible'`。公开接口不复用 `/api/admin/**`；点赞记录写入 `article_like`，文章点赞总数写入 `article.like_count`，两次写入由公共 Service 事务保证一致。

## config

Config 放配置类：

```text
SwaggerConfig
WebMvcConfig
MyBatisConfig
CorsConfig
```

拦截器注册、跨域、Swagger、Jackson、MyBatis 等全局配置都放这里。Swagger 需要配置 Bearer Token 认证方案，方便在接口文档页面统一填写登录接口返回的 JWT。Swagger/OpenAPI 文档展示不按字母顺序打乱字段，由全局配置统一按 Java 类字段声明顺序重排 Schema，避免每个 VO 重复手写字段顺序；运行时 JSON 返回也由全局 Jackson 配置关闭字段字母排序，避免 Swagger 执行接口时看到的响应体顺序和代码声明顺序不一致。拦截器本体不放在 `config`，放在 `interceptor`。

文章阶段学习 AOP 时，可以在 `springboot-article` 内放文章专用切面，例如 `ArticleLogAspect`。该切面只服务文章模块学习横切能力，不设计通用审计模块，不写入操作日志表。

## interceptor

Interceptor 放请求拦截器：

```text
AuthInterceptor
```

JWT 登录校验、请求放行判断、当前登录管理员 ID 写入请求上下文等拦截逻辑放这里。`WebMvcConfig` 只负责注册拦截器和配置拦截路径、白名单，不写具体 Token 校验逻辑。

## utils

Utils 放第三方工具封装和自定义工具类：

```text

PasswordUtils
DateTimeUtils
IpUtils
StringUtils
ImageUtils
```

规则：

- 工具类尽量无状态。
- 不依赖 Controller。
- 不直接写数据库。
- 不承载业务规则。

## 分页规范

分页使用 PageHelper，不在 Mapper XML 中手写通用分页。

分页能力用于文章、说说、相册、评论、留言、友链等集合型资源。

管理员模块属于单人后台场景，不设计管理员列表分页接口。

分页参数：

| 参数 | 规则 |
|---|---|
| page | 默认 1，不能小于 1 |
| size | 默认 10，不能小于 1，不能大于 100 |
| keyword | 可空 |
| status | 可空 |
| sort | 只能使用后端白名单字段 |
| order | 只能是 asc 或 desc |

分页返回统一包装为 `PageResult`：

```text
page
size
total
pages
list
```

少量固定数量查询，例如最新 5 条评论、热门 10 篇文章，可以不用分页插件。
