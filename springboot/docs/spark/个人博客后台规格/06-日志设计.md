# 06-日志设计

## 日志目标

日志只服务当前个人博客后台的开发和排查：

- 控制台能看到必要的运行日志。
- 业务代码可以通过 `@Slf4j` 输出 info、warn、error。
- 数据库 SQL 排查走 MyBatis 自带日志开关。
- 不做操作日志入库。
- 不做日志文件落地。
- 不额外引入日志系统。

当前后台只有单人使用，不需要记录“谁在什么时间做了什么操作”的审计日志。

## 日志分类

| 类型 | 用途 | 实现方式 |
|---|---|---|
| 控制台运行日志 | 排查启动、接口、业务异常 | Logback + SLF4J + Lombok `@Slf4j` |
| SQL 调试日志 | 排查数据库 SQL 执行情况 | MyBatis 日志开关 |
| 文章 AOP 行为日志 | 学习横切能力、观察文章关键动作和接口耗时 | Spring AOP + SLF4J 控制台输出 |

## 日志依赖

使用 Spring Boot 默认日志体系：

- Logback：实际日志实现。
- SLF4J：日志门面。
- Lombok `@Slf4j`：生成 `log` 对象。

不额外引入其他日志依赖。

业务类需要打印日志时使用：

```java
@Slf4j
```

示例：

```java
log.info("接口调用成功，adminId={}", adminId);
log.warn("参数可能不符合预期，username={}", username);
log.error("系统异常", e);
```

## 控制台运行日志

建议记录：

- 项目启动关键状态。
- 参数校验失败。
- 数据库操作异常。
- 未捕获异常。
- 重要业务状态变更。

日志级别：

| 级别 | 用途 |
|---|---|
| info | 正常业务事件 |
| warn | 可疑但不致命的问题 |
| error | 系统异常 |
| debug | 开发阶段临时排查 |

规则：

- 正常流程少打日志，避免控制台刷屏。
- 异常日志使用 `log.error("说明", e)`，保留堆栈。
- 不记录密码、Token、数据库口令等敏感信息。
- 不在业务代码里手写 SQL 日志。

## SQL 调试日志

数据库日志使用 MyBatis 自带开关，不引入额外日志组件。

所有 MyBatis Mapper 的 SQL 调试日志统一通过全局配置开启，不在单个 `Mapper.xml` 中单独实现日志。

当前项目所有 Mapper，包括 `AdminMapper.xml` 以及后续文章、分类、标签、说说、相册、评论、留言、友链、站点配置等 Mapper，都遵守同一规则：

- SQL 执行语句由 MyBatis 输出。
- SQL 参数由 MyBatis 输出。
- SQL 查询结果数量由 MyBatis 输出。
- 业务代码不重复打印 SQL。
- `Mapper.xml` 只写 SQL 映射和必要注释，不写日志逻辑。
- 排查结束后可以关闭 SQL 日志，避免控制台输出过多。

需要查看 MyBatis SQL 时，可以临时打开：

```yaml
mybatis:
  configuration:
    log-impl: org.apache.ibatis.logging.stdout.StdOutImpl
```

排查结束后可以关闭，避免控制台输出过多。

说明：

- SQL 日志属于 MyBatis 的职责。
- 业务代码不重复打印 SQL。
- 不单独设计数据库日志文件。

## Logback 控制台配置

第二阶段开始可以新增 `src/main/resources/logback-spring.xml`，只做业务控制台日志配置：

- 输出业务日志。
- 不接管 MyBatis SQL 调试日志。
- 不写入日志文件。
- 不配置日志滚动。
- 不记录密码、Token、数据库口令等敏感信息。

推荐的控制台配置：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>

    <property name="CONSOLE_LOG_PATTERN"
              value="%d{yyyy-MM-dd HH:mm:ss.SSS} %-5level [%thread] %logger{36} - %msg%n"/>

    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>${CONSOLE_LOG_PATTERN}</pattern>
            <charset>UTF-8</charset>
        </encoder>
    </appender>

    <logger name="com.xinmengqaq.springboot" level="INFO"/>

    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
    </root>

</configuration>
```

说明：

- `com.xinmengqaq.springboot` 控制业务代码日志。
- MyBatis SQL 日志继续由 `mybatis.configuration.log-impl` 控制。
- `Mapper.xml` 不需要添加日志配置。

## 不做的日志能力

当前项目不需要：

- `operation_log` 表。
- 操作日志入库。
- `operation.log` 操作日志文件。
- `app.log` 运行日志文件。
- `error.log` 错误日志文件。
- `access.log` 访问日志文件。
- 日志文件按天滚动、切分和保留策略。

## 关于 AOP

AOP 本身不是问题，但当前日志设计不因为后台审计或操作日志入库而引入 AOP。

文章模块进入 Spring Boot AOP 学习阶段后，可以实现文章专用切面：

- 只拦截文章新增、修改、删除、发布、隐藏等文章动作。
- 可以记录动作类型、文章 ID、执行结果和耗时到控制台。
- 可以记录文章详情、分页查询的耗时。
- 不写入数据库。
- 不提供操作日志查询接口。
- 不设计通用审计链路。
- 不吞异常，异常继续交给全局异常处理。

## 第一阶段要求

第一阶段只确认：

- 项目使用 Spring Boot 默认 Logback。
- 业务代码可以使用 Lombok `@Slf4j`。
- SQL 调试可以通过 MyBatis 日志开关打开。

第一阶段不新增：

- `logback-spring.xml`。
- `operation_log` 表。
- `OperationLogService`。
- `OperationLog` 注解。
- `OperationLogAspect`。

文章 AOP 学习阶段可以新增文章专用注解和切面，例如 `ArticleActionLog`、`ArticleLogAspect`，但命名和职责必须限定在文章模块内。
